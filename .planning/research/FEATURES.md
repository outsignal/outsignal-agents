# Feature Landscape: Client Portal Inbox

**Domain:** Unified inbox with email reply + LinkedIn messaging for B2B outbound client portal
**Researched:** 2026-03-11
**Confidence:** HIGH

---

## Existing Features (Out of Scope for v5.0 Research)

Already shipped and not re-researched:
- Read-only replies feed at `/portal/replies` (flat card list, no threading, no send)
- Reply model with: bodyText, intent, sentiment, objectionSubtype, classificationSummary, outboundSubject, outboundBody, campaignName, sequenceStep, senderEmail, senderName, emailBisonReplyId
- EmailBison webhook handling (LEAD_REPLIED, LEAD_INTERESTED, UNTRACKED_REPLY_RECEIVED)
- AI reply suggestion generation via Writer Agent (fires on webhook, sends to Slack)
- LinkedIn Voyager worker: sendMessage(), sendConnectionRequest(), profile_view, connect
- LinkedInAction queue with enqueueAction() — production-tested
- Workspace package field: email | linkedin | email_linkedin | consultancy
- Slack + email notifications for replies (17 types, audit logging)

## Table Stakes

Features every inbox must have. Without these, replacing `/portal/replies` adds no value.

| Feature | Why Expected | Complexity | Dependencies |
|---------|--------------|------------|--------------|
| **Two-panel inbox layout** | Standard inbox UX (thread list left, conversation right). Clients need to scan threads quickly and drill into one. | Low | None — CSS layout |
| **Email thread view** | Group replies by parent_id chain into threads. Show outbound email + all replies chronologically. | Medium | EmailBison parent_id field on replies |
| **Thread detail with message bubbles** | Inbound left-aligned, outbound right-aligned. Timestamp, sender name, body. | Low | Thread API route |
| **Unread indicators** | Dot/badge on unread threads. Unread count in nav. | Low | readAt field on Reply model (null = unread) |
| **Intent/sentiment badges** | Show existing AI classification on inbound messages. Already stored on Reply model. | Low | Existing Reply.intent + Reply.sentiment |
| **Channel tabs** | Email / LinkedIn / All tabs based on workspace package. | Low | Workspace.package field |
| **Inbox nav item** | Replace "Replies" with "Inbox" in portal sidebar. | Low | Portal sidebar component |

## Differentiators

Features that go beyond basic inbox and leverage existing Outsignal infrastructure.

| Feature | Value Proposition | Complexity | Dependencies |
|---------|-------------------|------------|--------------|
| **Email reply from portal** | Clients can reply without leaving portal. Uses EmailBison `POST /replies/{id}/reply`. | Medium | EmailBison sendReply API (UNVERIFIED — needs spike) |
| **LinkedIn conversation view** | Full message history for LinkedIn threads. Fetched via Voyager API, stored in DB. | High | New VoyagerClient methods + DB models + worker endpoint |
| **LinkedIn reply queue** | Queue a LinkedIn message from portal. Uses existing LinkedInAction model with priority 1. | Low | Existing enqueueAction() — battle-tested |
| **AI suggested reply** | Pre-generated reply suggestion shown as "Use this" button. Already generated by Writer Agent at webhook time. | Low | suggestedReply field on Reply model (already generated, just needs display) |
| **Outbound context panel** | Show the original outbound email/LinkedIn message that triggered the reply. Already stored on Reply model (outboundSubject, outboundBody). | Low | Existing Reply fields |
| **Sender selection in composer** | Dropdown to pick which sender email to reply from. Workspace has multiple senders. | Low | Existing Sender model + EmailBison sender_email_id param |

## Anti-Features

Features to explicitly NOT build.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Real-time WebSockets/SSE** | Vercel serverless has no persistent connections. Reply volume is ~5-20/day — not enough to justify real-time infra. | Poll every 15s when inbox is open, 60s background. |
| **Rich text editor (Tiptap/ProseMirror)** | Rich HTML emails hurt deliverability for cold outbound. B2B replies should be plain text. | Plain textarea with auto-resize. |
| **Bulk reply** | Mass replying to cold outbound replies is dangerous — each needs individual attention. | One-at-a-time reply only. |
| **Email thread reconstruction from headers** | Parsing Message-ID/In-Reply-To headers to build threads is brittle. | Use EmailBison's parent_id field for threading. |
| **Unified search across channels** | Complex to implement cross-channel search (email body + LinkedIn messages). Low value at current volume. | Per-channel thread list is sufficient. Filter by channel tab. |
| **Attachment handling** | Sending attachments in cold outbound replies is rare and hurts deliverability. | Text-only replies. Show attachment indicators on inbound messages (read-only). |
| **Draft saving** | At current volume (~5-20 replies/day across all workspaces), auto-save drafts is over-engineering. | Just send. Composer state lives in React state. |

```
                                    |

                          |                         |
                          v                         v
                   Conversation view ──> AI suggestion display
                          |
                          v
                   Reply composer (email mode)

VoyagerClient extensions ──> Worker conversations endpoint
          |
          v
LinkedInConversation + LinkedInMessage models
          |
          v
LinkedIn sync API ──> LinkedIn conversation view
          |                    |
          v                    v
LinkedIn thread API ──> Reply composer (LinkedIn mode)

Sidebar nav update (independent, do last)
```

**Critical path:** EmailBison sendReply spike is the #1 risk. If the endpoint doesn't work, email reply falls back to mailto: deeplink (functional but less seamless).

**Parallel tracks:**
- Track A: Email thread API + inbox shell + email reply
- Track B: LinkedIn worker extensions + DB models + LinkedIn sync + LinkedIn view
- Track C (last): Nav update, polish, empty states

## Schema Changes Needed

### New Models

```prisma
model LinkedInConversation {
  id                String   @id @default(cuid())
  linkedinId        String   // LinkedIn's conversation entityUrn
  workspaceSlug     String
  senderId          String   // which Sender account fetched this
  participantName   String?
  participantUrn    String?
  participantUrl    String?  // LinkedIn profile URL
  personId          String?  // link to Person if matched
  lastActivityAt    DateTime?
  unreadCount       Int      @default(0)
  lastSyncedAt      DateTime?
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  messages          LinkedInMessage[]
  sender            Sender   @relation(fields: [senderId], references: [id])
  person            Person?  @relation(fields: [personId], references: [id])

  @@unique([linkedinId, senderId])
  @@index([workspaceSlug])
  @@index([senderId])
  @@index([personId])
}

model LinkedInMessage {
  id               String   @id @default(cuid())
  linkedinEventUrn String   // LinkedIn's event entityUrn
  conversationId   String
  senderUrn        String   // who sent this message
  senderName       String?
  body             String
  isOutbound       Boolean  // true if sent by our Sender
  createdAt        DateTime // LinkedIn's timestamp

  conversation     LinkedInConversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)

  @@unique([linkedinEventUrn])
  @@index([conversationId])
}
```

## MVP Recommendation

**Phase 1 (API Foundation):** EmailBison sendReply spike + client extensions + LinkedIn worker conversation fetch
**Phase 2 (Data Layer):** LinkedIn DB models + sync API
**Phase 3 (Email Inbox):** Thread API + inbox shell + conversation view + email reply composer
**Phase 4 (LinkedIn Inbox):** LinkedIn thread API + LinkedIn conversation view + LinkedIn reply composer
**Phase 5 (Polish):** Nav update, channel tabs, empty states, loading skeletons, error handling
