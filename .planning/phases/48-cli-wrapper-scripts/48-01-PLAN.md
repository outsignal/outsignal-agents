  <action>
1. Install tsup as devDependency: `npm install --save-dev tsup`

2. Create `scripts/cli/_cli-harness.ts` — the shared wrapper utility:
   - Import `sanitizeOutput` from `@/lib/sanitize-output`
   - Export `runWithHarness(usage: string, fn: () => Promise<unknown>): Promise<void>`
   - try/catch wrapping: success = `{ ok: true, data }`, failure = `{ ok: false, error, usage }`
   - Pretty-print with `JSON.stringify(..., null, 2)`
   - Apply `sanitizeOutput()` to the JSON string before writing to stdout
   - Write to `process.stdout.write(sanitized + "\n")`
   - Exit 0 on success, exit 1 on failure
   - The underscore prefix means tsup entry glob `scripts/cli/*.ts` with `!scripts/cli/_*.ts` excludes it from direct compilation (it gets inlined into each bundle)

3. Create `tsup.cli.config.ts` in project root:
   - entry: `["scripts/cli/*.ts", "!scripts/cli/_*.ts"]`
   - outDir: `"dist/cli"`
   - format: `["cjs"]`
   - bundle: true, splitting: false, clean: true
   - external: `["@prisma/client"]`
   - esbuildOptions: set `options.alias = { "@": path.resolve(__dirname, "src") }`
   - Use `import { defineConfig } from "tsup"` and `import path from "path"`

4. Add to package.json scripts: `"build:cli": "tsup --config tsup.cli.config.ts"`

5. Add `dist/cli/` to `.gitignore` (if not already present). Check for existing `dist/` entry first.

6. Ensure `dotenv` is importable. It should be a transitive dep already — verify with `ls node_modules/dotenv`. If missing, add to devDependencies.
  </action>
  <verify>
    npm run build:cli 2>&1 | tail -5
    # Should show tsup output with 0 errors. May show "no entry files found" warning since no non-underscore scripts exist yet — that's fine for this task.
  </verify>
  <done>tsup installed, cli-harness.ts created, tsup config created, build:cli script in package.json, dist/cli/ gitignored</done>

<task type="auto">
  <name>Task 2: Create workspace-get wrapper script, compile, and smoke-test end-to-end</name>
  <files>
    scripts/cli/workspace-get.ts
  </files>
  <action>
1. Create `scripts/cli/workspace-get.ts`:
   ```
   import { config } from "dotenv";
   config({ path: ".env" });
   config({ path: ".env.local" });

   import { runWithHarness } from "./_cli-harness";
   import { prisma } from "@/lib/db";

   const [,, slug] = process.argv;

   runWithHarness("workspace-get <slug>", async () => {
     if (!slug) throw new Error("Missing required argument: slug");
     const ws = await prisma.workspace.findUnique({ where: { slug } });
     if (!ws) throw new Error(`Workspace '${slug}' not found`);
     return ws;
   });
   ```

   NOTE: Do NOT import from `src/lib/agents/writer.ts` for workspace-get. The writer's `getWorkspaceIntelligence` tool does more than just workspace lookup (it also fetches websiteAnalysis). For the smoke test, use a direct Prisma query — this validates Prisma external + @/ alias + dotenv + harness + sanitization in one shot. The full tool function wrappers come in Plans 02/03.

   IMPORTANT: The orchestrator's `getWorkspaceInfo` tool (used by research + orchestrator agents) is a richer version that includes quota, modules, and package info. The CLI `workspace-get` script should match that richer version. Check `src/lib/agents/orchestrator.ts` for the `getWorkspaceDetails` function it calls internally, and replicate that in the wrapper. For the smoke test, a simple Prisma query is sufficient — Plan 02/03 will refine it.

2. Run `npm run build:cli` — verify `dist/cli/workspace-get.js` is created

3. Test success case: `node dist/cli/workspace-get.js rise`
   - Must return `{ "ok": true, "data": { ... } }` with workspace fields
   - Must NOT contain any DATABASE_URL or API keys in output

4. Test error case: `node dist/cli/workspace-get.js`
   - Must return `{ "ok": false, "error": "Missing required argument: slug", "usage": "workspace-get <slug>" }`
   - Must exit with code 1

5. Test invalid workspace: `node dist/cli/workspace-get.js nonexistent`
   - Must return `{ "ok": false, "error": "Workspace 'nonexistent' not found", "usage": "workspace-get <slug>" }`

6. If the `__dirname` / `PROJECT_ROOT` issue surfaces (load-rules.ts path resolution), fix by setting `process.env.PROJECT_ROOT = process.cwd()` at the top of `_cli-harness.ts` (before any imports that use it). This is the documented fix from load-rules.ts.
