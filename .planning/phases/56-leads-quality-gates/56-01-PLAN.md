Create `src/lib/discovery/quality-gate.ts` following the copy-quality.ts pattern (pure functions, typed results, no side effects).

**Interfaces to export:**
- `QualityMetrics`: totalResults, verifiedEmailCount, verifiedEmailPct, linkedinUrlCount, linkedinUrlPct, icpFitDistribution ({ high, medium, low, none }), junkCount, junkExamples (up to 5 strings), belowThreshold (true if verifiedEmailPct < 50)
- `QualityReport`: metrics, grade ("good" | "acceptable" | "low" | "poor"), suggestions (string[]), costPerVerifiedLead (number | null)

**Functions to export:**
1. `detectJunk(person: DiscoveredPersonResult): boolean` — returns true if:
   - Email starts with junk prefixes: info@, admin@, support@, sales@, contact@, hello@, noreply@, no-reply@, webmaster@, office@
   - Email contains @discovery.internal (placeholder from promotion.ts)
   - Name matches junk patterns: single char, "N/A", "Unknown", "Test", "None", "null" (case insensitive)
   - Both firstName AND lastName are missing/empty
   - Both email AND linkedinUrl are missing/null

2. `assessSearchQuality(people: DiscoveredPersonResult[], options?: { costUsd?: number; workspaceIcp?: { titles?: string[]; locations?: string[]; industries?: string[] } }): QualityReport`
   - Compute all 4 metrics from the DiscoveredPersonResult array
   - For ICP fit: lightweight string matching — match jobTitle against workspaceIcp.titles (case-insensitive contains), location against workspaceIcp.locations, company industry signal from companyDomain. Label as "preliminary ICP fit". If no workspaceIcp provided, all are "none".
   - Grade: "good" (>70% verified email + <5% junk), "acceptable" (50-70% verified email), "low" (30-50% verified email), "poor" (<30%)
   - Suggestions: generate actionable suggestions based on metrics. E.g. "Verified email rate 22%. Suggest: run enrichment waterfall on remaining 78%, or try different filters." Include routing suggestion for unverified emails.
   - costPerVerifiedLead: costUsd / verifiedEmailCount (null if no cost data or zero verified)

CRITICAL PITFALL: DiscoveredPersonResult is the STAGING table shape — it does NOT have enrichmentData. Check email field directly (non-null AND not placeholder @discovery.internal). "Verified email" at staging time = has a real email address (not placeholder, not junk prefix).

**Test file** `src/lib/discovery/__tests__/quality-gate.test.ts`:
- Test detectJunk with junk emails (info@, admin@), placeholders (@discovery.internal), empty names, valid people
- Test assessSearchQuality with a mixed array: some with emails, some without, some junk. Verify all 4 metrics computed correctly.
- Test grade thresholds: >70% = good, 50-70% = acceptable, 30-50% = low, <30% = poor
- Test costPerVerifiedLead calculation (including null when no cost)
- Test ICP fit distribution with and without workspaceIcp
- At least 10 test cases
