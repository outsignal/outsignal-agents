```typescript
// --- Validator Agent Types (Phase 55) ---

export const validationFindingSchema = z.object({
  check: z.enum([
    "filler_spintax",
    "tonal_mismatch",
    "angle_repetition",
    "ai_patterns",
    "structural",
    "general",
  ]),
  severity: z.enum(["hard", "soft"]),
  step: z.number().optional(),
  field: z.string().optional(),
  problem: z.string(),
  suggestion: z.string(),
});

export const validationResultSchema = z.object({
  passed: z.boolean(),
  findings: z.array(validationFindingSchema),
  summary: z.string(),
  checklist: z.object({
    fillerSpintax: z.enum(["pass", "fail", "warn"]),
    tonalMismatch: z.enum(["pass", "fail", "warn"]),
    angleRepetition: z.enum(["pass", "fail", "warn"]),
    aiPatterns: z.enum(["pass", "fail", "warn"]),
  }),
});

export type ValidationFinding = z.infer<typeof validationFindingSchema>;
export type ValidationResult = z.infer<typeof validationResultSchema>;
```

**validator-rules.md** -- Create `.claude/rules/validator-rules.md` with the full validator prompt. Structure:

1. **Purpose section**: You are the third quality gate. Writer self-reviews first (gate 1), writer cross-step dedup second (gate 2), you are gate 3. Your job: semantic analysis that only an LLM can assess. Structural checks are handled deterministically by the wrapper -- focus on what code cannot catch.

2. **Severity mapping**: hard = must fix before save (writer will attempt one rewrite). soft = save with flag (admin sees in review UI). Be specific about which checks map to which severity.

3. **Checklist (4 mandatory sections)**:
   - **Filler Spintax**: Options are interchangeable throwaways with no substantive difference. Give 3 bad examples ({just a thought|one more thing}, {meant to ask|been meaning to say}, {quick one|genuine question}) and 3 good examples (different value props, different proof points). Severity: hard for clear filler, soft for borderline.
   - **Tonal Mismatch**: Copy doesn't match the provided outreachTonePrompt or feels inconsistent across steps. Check: does the overall voice match the tone directive? Do steps feel like they were written by the same person? Severity: hard for gross mismatch, soft for slight drift.
   - **Angle Repetition Across Steps**: Same value prop or pain point reused in multiple steps. Each step should bring a NEW angle, proof point, or hook. Severity: hard for identical angle restated, soft for similar but distinguishable.
   - **AI-Sounding Patterns**: Overly structured sentences, unnatural transitions, template-feeling copy that technically passes banned phrase lists. Give 3 examples of natural copy that passes and 3 examples of AI-sounding copy that should be flagged. Severity: ALWAYS soft (inherently subjective).

4. **General Observations**: After the checklist, note any other quality issues. Weight: always soft severity. This is the open-ended section for novel catches.

5. **Review Philosophy**: Balanced strictness. Flag clear issues, let borderline cases through. If you would flag 5+ issues on a sequence, reconsider -- you may be too strict. The goal is catching real problems, not creating alert fatigue.

6. **Output Format**: Return ONLY a JSON object matching the ValidationResult schema. No markdown, no explanation, no code fences. Just raw JSON. Set `passed: true` if zero hard findings, `passed: false` if any hard findings. Checklist values: "pass" if no issues, "fail" if hard issue found, "warn" if soft issue found.

Create `scripts/cli/validate-sequence.ts` following the _cli-harness pattern used by all other scripts in scripts/cli/.

The wrapper does THREE things:
1. Reads input JSON file (--file argument) containing `{ steps: [...], context: { vertical, outreachTonePrompt, icpIndustries, icpDecisionMakerTitles, strategy } }`
2. Runs DETERMINISTIC structural checks using copy-quality.ts functions on each step (checkCopyQuality on body/subject/subjectVariantB, checkWordCount, checkGreeting on step 1, checkCTAFormat, checkSubjectLine on subjects, checkLinkedInSpintax for LinkedIn steps, checkSequenceQuality on all steps as a unit). Collect structural findings as ValidationFinding objects with check: "structural".
3. Invokes Claude Code CLI for SEMANTIC analysis: builds a prompt containing the sequence steps + workspace context + instruction to return ValidationResult JSON. Uses `execSync` with `claude -p "{prompt}" --output-format json` (NOT --skill flag since .claude/skills/ is empty and untested). Set 60s timeout.
4. Parses the Claude output: try raw JSON.parse first, then regex for ```json code blocks, then regex for JSON object in surrounding text. Validate with validationResultSchema.safeParse.
5. MERGES structural findings into the semantic result: append structural findings to the findings array, update passed to false if any structural finding is hard severity.
6. On timeout or parse failure: return a safe fallback `{ passed: true, findings: structuralFindings, summary: "Semantic validation skipped (timeout/parse error) -- structural checks only", checklist: { fillerSpintax: "pass", tonalMismatch: "pass", angleRepetition: "pass", aiPatterns: "pass" } }` -- so structural findings still surface even if LLM fails.

**Prompt construction** for the Claude CLI call:
- Include the full sequence steps as JSON
- Include workspace context (vertical, outreachTonePrompt, strategy, ICP summary)
- Reference validator-rules.md: "Follow the rules in .claude/rules/validator-rules.md"
- End with: "Return ONLY a raw JSON object matching the ValidationResult schema. No markdown, no explanation."
- Keep the prompt focused -- do NOT include the full writer-rules.md (260+ lines would bloat context)
