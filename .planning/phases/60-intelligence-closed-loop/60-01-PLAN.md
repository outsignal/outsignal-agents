  <files>src/lib/outbound-copy-lookup.ts, scripts/backfill-outbound-copy.ts</files>
  <action>
Create `src/lib/outbound-copy-lookup.ts` with a single exported async function:

```typescript
export async function lookupOutboundCopy(
  campaignId: string,          // Outsignal campaign ID (UUID)
  sequenceStep: number | null, // 1-indexed position from EB webhook
): Promise<{ subject: string | null; body: string | null }>
```

Implementation:
1. Look up Campaign by `id`, select `emailSequence` and `emailBisonCampaignId`
2. Try local emailSequence first (fast, no API call): parse JSON, find step by position match
3. If local fails (null emailSequence), fall back to EB API:
   - Find the workspace that owns this campaign (via `prisma.workspace.findFirst` with campaign relation)
   - Get workspace apiToken
   - Instantiate EmailBisonClient with the token
   - Call `getSequenceSteps(emailBisonCampaignId)`
   - If sequenceStep is not null: match by position (try exact match, then try position-1 for off-by-one)
   - If sequenceStep is null but only 1 step exists: use that step
4. Return `{ subject: null, body: null }` as fallback

Create `scripts/backfill-outbound-copy.ts`:
1. Load dotenv (.env then .env.local)
2. Query all replies WHERE campaignId IS NOT NULL AND outboundSubject IS NULL (using Outsignal campaignId, not EB campaignId)
3. Group replies by campaignId to minimize API calls (Map<campaignId, Reply[]>)
4. For each campaign group:
   - Call lookupOutboundCopy for the first reply to cache the EB API result
   - Actually, refactor: create a cache Map<campaignId, SequenceStep[]> to avoid repeated API calls
   - For each reply: match by sequenceStep, update outboundSubject + outboundBody via prisma.reply.update
5. Add a small delay (200ms) between campaign groups to avoid rate limiting
6. Report per-workspace: "Rise: 85/105 populated, 20 skipped (no sequenceStep match)"
7. Report totals: "Total: 320/407 populated, 87 skipped"

Run with: `npx tsx scripts/backfill-outbound-copy.ts`

Note: The lookup utility needs to handle the campaignId correctly. In process-reply.ts, the `campaignId` from the webhook payload is the EB campaign ID (string of integer). But in the Reply model, `campaignId` is also stored as this EB campaign ID string. The lookup function should accept the Outsignal campaign UUID (from Reply.campaign relation) -- but actually Reply stores both: Reply.campaignId (EB string) and Reply.campaign (relation to Outsignal Campaign). Check the Reply model to see which field to use. The backfill script should group by Outsignal campaign relation (since that has emailBisonCampaignId for the API call).

For the cache approach: create a helper that fetches and caches sequence steps per emailBisonCampaignId, so the main lookupOutboundCopy function can also use it. This avoids 25 x N API calls when N replies share the same campaign.
