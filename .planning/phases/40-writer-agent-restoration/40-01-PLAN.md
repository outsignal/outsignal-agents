Create `trigger/generate-suggestion.ts` with the following structure:

**Imports:**
- `task` from `@trigger.dev/sdk`
- `PrismaClient` from `@prisma/client` (module-scope instance, same pattern as process-reply.ts)
- `anthropicQueue` from `./queues`
- `runAgent` from `@/lib/agents/runner`
- `writerConfig` from `@/lib/agents/writer`
- `postMessage` from `@/lib/slack`

**Payload interface** — `GenerateSuggestionPayload`:
```typescript
export interface GenerateSuggestionPayload {
  replyId: string;       // cuid from Reply table
  workspaceSlug: string;
}
```

**Task definition** — `generate-suggestion`:
- `id: "generate-suggestion"`
- `queue: anthropicQueue`
- `maxDuration: 300` (5 minutes — Opus + KB search + multi-step tool calls can take 60-120s, plus margin)
- `retry: { maxAttempts: 2, factor: 2, minTimeoutInMs: 5_000, maxTimeoutInMs: 30_000 }` (only 2 attempts — Opus calls are expensive)

**Run function** — 4 steps:

**Step 1: Load Reply + thread context from DB**
- Fetch the Reply record by `replyId` using prisma
- If not found, log error and return early with `{ success: false, reason: "reply-not-found" }`
- If `reply.aiSuggestedReply` is already populated, log and return early with `{ success: false, reason: "already-generated" }` (idempotency guard)
- Load the workspace record to get `slackChannelId`
- Build thread context: if reply has `emailBisonParentId`, fetch sibling replies in the same thread (same `emailBisonParentId` or same `leadEmail` + `workspaceSlug`, ordered by `receivedAt`) to give the writer full conversation context

**Step 2: Run writer agent**
- Build the user message for the writer's Reply Suggestion Mode. The message MUST start with "suggest reply" to activate reply mode (per writer.ts system prompt). Format:

```
suggest reply to inbound email

Workspace: {workspaceSlug}

Lead: {reply.senderName ?? reply.senderEmail}
Email: {reply.senderEmail}
Subject: {reply.subject ?? "(no subject)"}
Classification: {reply.intent ?? "unknown"} / {reply.sentiment ?? "unknown"}
Campaign: {reply.campaignName ?? "unknown"}

Their message:
{reply.bodyText}

{if outboundSubject/outboundBody exist:}
Original outbound email they replied to:
Subject: {reply.outboundSubject}
Body: {reply.outboundBody}

{if thread context exists:}
Thread history (oldest first):
{for each sibling reply: "[{direction}] {senderName}: {bodyText}" }
```

- Call `runAgent(writerConfig, userMessage, { triggeredBy: "trigger-task", workspaceSlug })`
- The writer agent returns structured WriterOutput, but in reply mode it returns plain prose. Extract the suggestion text from `result.text`.
- IMPORTANT: The writer in reply mode returns plain text (not JSON). Use `result.text` directly as the suggestion. Do NOT try to parse JSON from it.

**Step 3: Persist aiSuggestedReply**
- Update the Reply record: `prisma.reply.update({ where: { id: replyId }, data: { aiSuggestedReply: suggestion } })`
- Log success with reply ID and suggestion length

**Step 4: Send Slack follow-up**
- Format: `*AI Suggested Reply for {senderName || senderEmail}:*\n{suggestion}`
- Post to workspace's slackChannelId (if set)
- Post to `process.env.REPLIES_SLACK_CHANNEL_ID` (if set)
- Both Slack calls wrapped in `.catch(() => {})` — non-blocking

**Return value:**
```typescript
return {
  success: true,
  replyId,
  suggestionLength: suggestion.length,
  durationMs: result.durationMs,
};
```

**Error handling:** Wrap the entire run in try/catch. On error, log `[generate-suggestion] Failed for reply ${replyId}:` and re-throw (Trigger.dev retry will handle it). Do NOT catch and swallow — let retry logic work.

**Key constraints (from CONTEXT.md decisions):**
- Zero changes to `src/lib/agents/` framework files
- Use `runAgent` + `writerConfig` as-is — the writer agent already has Reply Suggestion Mode built in
- Model is Opus (configured in writerConfig) — this is the whole point of the migration
