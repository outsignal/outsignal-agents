## Summary

Phase 35 builds the email inbox for the client portal: thread list, conversation view, reply composer, and AI suggestion display. The portal infrastructure (auth, sidebar, session management) is fully built. The EmailBison client already has `sendReply()`, `getReply()`, and `getRepliesPage()` validated in Phase 33. The core challenge is that the current `Reply` Prisma model is missing several fields required for threading, HTML display, sending, and AI suggestions.

**The most important planning discovery:** The `Reply` model needs a schema migration to add `emailBisonParentId` (Int?), `leadEmail` (String?), `htmlBody` (Text?), `ebSenderEmailId` (Int?), `interested` (Boolean), and `aiSuggestedReply` (String?). Without these fields, the thread grouping API (EMAIL-01), HTML email rendering (EMAIL-02), auto-sender-select on reply (EMAIL-03), and AI suggestion display (EMAIL-04) are all blocked. The webhook handler and poll-replies cron also need updates to populate these new fields.

The phase should be structured in three waves: (1) DB migration + data layer, (2) API routes for thread list and thread detail, (3) portal UI with composer and AI suggestion panel. The sidebar nav change (replace "Replies" with "Inbox") is a small final task.

**Primary recommendation:** Lead with the schema migration. Every subsequent task depends on `emailBisonParentId` being available to build the threading logic.

### Pattern 2: Thread Grouping Algorithm

**What:** Group EmailBison replies into visual threads using `parent_id` chain. The EB Reply API returns all messages (inbound + outbound "Outgoing Email" folder) with `parent_id` linking replies to their parent.
**When to use:** Thread list API and thread detail API.

```typescript
// Thread grouping: replies share a thread if they chain to the same root
// A "root" is any reply where parent_id is null OR parent_id points to a non-existent reply
// Key insight from EB data: outbound "Sent" folder items have type="Outgoing Email", folder="Sent"
// Inbound replies have type="Tracked Reply", folder="Inbox"

type ThreadRoot = {
  threadId: number;        // emailBisonParentId of the root (or emailBisonReplyId if root itself)
  leadEmail: string;
  subject: string | null;
  lastMessageAt: Date;
  lastSnippet: string;
  messageCount: number;
  interested: boolean;
  replyStatus: 'awaiting_reply' | 'replied' | 'new';
  aiSuggestedReply: string | null;
}

function buildThreadId(reply: Reply): number {
  // Thread root = the reply with no parent, or the parent that doesn't exist locally
  return reply.emailBisonParentId ?? reply.emailBisonReplyId!;
}
```

**Threading logic:** All replies sharing the same `threadId` (root of the chain) belong to one thread. Orphaned parents (replies whose parent_id points to an ID not in DB) are treated as roots.

### Pattern 3: Thread List API

**What:** GET `/api/portal/inbox/email/threads` — returns grouped thread summaries for the workspace.

```typescript
// Source: project pattern (portal-session + prisma query)
export async function GET(request: Request) {
  const { workspaceSlug } = await getPortalSession();

  // Fetch all inbox replies for workspace
  const replies = await prisma.reply.findMany({
    where: {
      workspaceSlug,
      // Only inbound replies have content we want to thread
      // Outbound sent messages are fetched via EB API for context only
    },
    orderBy: { receivedAt: 'desc' },
    take: 200, // Cap at 200 most recent for performance
  });

  // Group into threads by threadId (emailBisonParentId ?? emailBisonReplyId)
  const threadMap = new Map<number, Reply[]>();
  for (const reply of replies) {
    const threadId = reply.emailBisonParentId ?? reply.emailBisonReplyId!;
    const group = threadMap.get(threadId) ?? [];
    group.push(reply);
    threadMap.set(threadId, group);
  }

  // Build thread summary from each group
  const threads = Array.from(threadMap.entries()).map(([threadId, msgs]) => {
    const sorted = msgs.sort((a, b) => b.receivedAt.getTime() - a.receivedAt.getTime());
    const latest = sorted[0];
    // replyStatus: look at the folder/type of latest message
    // ... see Anti-Patterns below for replyStatus derivation
    return { threadId, latest, msgCount: msgs.length };
  });

  // Sort threads by most recent activity
  threads.sort((a, b) => b.latest.receivedAt.getTime() - a.latest.receivedAt.getTime());

  return NextResponse.json({ threads });
}
```

### Pattern 7: HTML Email Sandboxing

**What:** HTML email bodies must be rendered safely. The decision is to use a sandboxed `<iframe>` with `srcdoc`.

```tsx
// Recommended approach: iframe with srcdoc + sandbox attribute
// This prevents JS execution, form submission, and same-origin access
<iframe
  srcDoc={htmlBody}
  sandbox="allow-popups allow-popups-to-escape-sandbox"
  className="w-full min-h-[120px] border-0"
  onLoad={(e) => {
    // Auto-resize iframe to content height
    const iframe = e.currentTarget;
    iframe.style.height = `${iframe.contentDocument?.body.scrollHeight ?? 120}px`;
  }}
/>
```

**Why `sandbox` with `allow-popups`:** The empty `sandbox` attribute blocks everything including links. `allow-popups` lets email links open in a new tab (expected UX). Do NOT add `allow-scripts` — cold email HTML should never execute JS.

### Pattern 8: Polling for New Messages (Portal UI)

**What:** Portal inbox page polls for new threads while the user is active. Uses existing project pattern (no WebSockets/SSE — Vercel serverless).

```typescript
// Client component polling: useEffect with setInterval
// 15s when tab is active (document.visibilityState === 'visible')
// 60s when tab is hidden
// Per STATE.md decision: "Polling (15s active, 60s background) not WebSockets/SSE"

useEffect(() => {
  const refresh = () => fetchThreads();
  refresh(); // initial load

  const getInterval = () =>
    document.visibilityState === 'visible' ? 15_000 : 60_000;

  let timer = setInterval(refresh, getInterval());

  const handleVisibility = () => {
    clearInterval(timer);
    timer = setInterval(refresh, getInterval());
  };
  document.addEventListener('visibilitychange', handleVisibility);
  return () => {
    clearInterval(timer);
    document.removeEventListener('visibilitychange', handleVisibility);
  };
}, []);
```

### Anti-Patterns to Avoid

- **Thread by lead email only (not parent_id):** Same lead can be in multiple campaigns — grouping purely by email conflates unrelated conversations. Use `parent_id` chain as primary grouping key.
- **Fetch all EB replies on every request:** `getReplies()` fetches all pages — too slow for an inbox. Use local DB (`Reply` table) as the data source; it is kept current by webhook + poll-replies cron.
- **Optimistic send:** The UI decision is "wait for API confirmation before showing sent message." Do NOT add the sent message to the thread until `sendReply()` returns success.
- **Re-fetching full EB reply list to build thread:** The portal API should query the local `Reply` table, not call EmailBison API directly on every page load. EB API is the source of truth for new messages only (via webhook/poll).
- **Storing aiSuggestedReply as a separate model:** Keep it as a nullable string field on Reply — there's at most one suggestion per inbound reply, and no versioning needed.

## Common Pitfalls

### Pitfall 1: Missing `emailBisonParentId` on polled replies

**What goes wrong:** The `poll-replies` cron creates Reply records but currently does NOT store `parent_id` from the EB Reply object. Threads built from polled replies will appear as orphaned singles.
**Why it happens:** The cron was built before threading was a requirement.
**How to avoid:** Update the poll-replies upsert to include `emailBisonParentId: reply.parent_id ?? null` in the create data.
**Warning signs:** Thread list shows every reply as a separate thread (no grouping).

### Pitfall 2: Missing `aiSuggestedReply` for existing Reply records

**What goes wrong:** The schema migration adds `aiSuggestedReply` but historical Reply records have null. The AI suggestion card won't show for any old replies (expected), but also won't show for new replies if the webhook handler isn't updated simultaneously.
**Why it happens:** Webhook update is a separate code path from schema migration.
**How to avoid:** Update webhook handler in the same plan as the schema migration — they're coupled.
**Warning signs:** AI suggestion card never appears in portal even after new replies come in.

### Pitfall 3: Outbound "Sent" replies in EB are separate records

**What goes wrong:** When the portal sends a reply via `sendReply()`, EmailBison creates a new Reply object with `folder="Sent"`, `type="Outgoing Email"`. If the poll-replies cron fetches these, they appear as unrelated inbound messages.
**Why it happens:** EB doesn't distinguish outbound records explicitly in the local model.
**How to avoid:** Filter the thread detail view by `Reply.emailBisonParentId` — the sent reply will have `parent_id` pointing back to the inbound reply. Store sent replies with a local `direction` or use the Reply `folder` field. Alternatively: when the portal sends a reply, create a local DB record immediately using the EB response (`result.data.reply`) so it appears instantly without waiting for the poll cron.
**Warning signs:** Sent messages appear twice in the conversation view (once from portal optimistic write, once from poll).

### Pitfall 4: `sendReply` requires `ebSenderEmailId` but current Reply model doesn't store it

**What goes wrong:** The auto-select sender flow breaks because `Reply.ebSenderEmailId` is null on existing records.
**Why it happens:** `sender_email_id` from the EB Reply object was never persisted.
**How to avoid:** Add `ebSenderEmailId` to the schema and update webhook + poll routes to store `reply.sender_email_id`. For the backfill of existing records, a migration script can re-fetch from EB if needed (or accept nulls on historical data and show an error state in the UI).
**Warning signs:** Reply composer send button fails immediately with "sender not found."

### Pitfall 5: HTML iframe height doesn't auto-resize

**What goes wrong:** `<iframe srcdoc>` defaults to a fixed height, truncating long emails.
**Why it happens:** iframes don't auto-resize to content by default due to cross-origin restrictions. Even `srcdoc` in sandbox mode restricts access to `contentDocument`.
**How to avoid:** Set `sandbox="allow-same-origin allow-popups"` for srcdoc iframes (srcdoc is considered same-origin when sandbox includes allow-same-origin), then use the `onLoad` handler to read `iframe.contentDocument.body.scrollHeight`.
**Warning signs:** Emails appear cut off; no scroll inside iframe.

### Pitfall 6: Thread list `replyStatus` derivation is ambiguous

**What goes wrong:** Status ("awaiting reply", "replied", "new") requires knowing which message is "ours" (outbound) vs the lead's (inbound). EB's `folder` field ("Inbox" vs "Sent") is the source of truth, but polled records may not have it stored locally.
**Why it happens:** Current Reply model has no `folder` or `direction` field.
**How to avoid:** Add a `direction` field to Reply model (`inbound` | `outbound`, default `inbound`). Set to `outbound` when: (a) created by `sendReply()` response, or (b) Reply.folder === "Sent" in the EB type. Use this for thread status derivation.
**Warning signs:** All threads show "awaiting reply" even after client has sent a reply.

## Code Examples

### Thread Grouping (Pure TypeScript)

```typescript
// Source: project pattern — no library needed at this data volume
type GroupedThread = {
  threadId: number;
  messages: Reply[];
  latestAt: Date;
};

function groupRepliesIntoThreads(replies: Reply[]): GroupedThread[] {
  const map = new Map<number, Reply[]>();

  for (const reply of replies) {
    // Determine thread root: parent_id if set, else this reply's own EB id
    const threadId = reply.emailBisonParentId ?? reply.emailBisonReplyId!;
    const bucket = map.get(threadId) ?? [];
    bucket.push(reply);
    map.set(threadId, bucket);
  }

  return Array.from(map.entries())
    .map(([threadId, messages]) => ({
      threadId,
      messages: messages.sort((a, b) => a.receivedAt.getTime() - b.receivedAt.getTime()),
      latestAt: messages.reduce((max, m) => m.receivedAt > max ? m.receivedAt : max, messages[0].receivedAt),
    }))
    .sort((a, b) => b.latestAt.getTime() - a.latestAt.getTime());
}
```

### Portal Reply Send API

```typescript
// POST /api/portal/inbox/email/reply
// Source: EmailBison spike results from Phase 33 (STATE.md [33-01])
export async function POST(request: Request) {
  const { workspaceSlug } = await getPortalSession();
  const { replyId, message } = await request.json() as { replyId: string; message: string };

  const replyRecord = await prisma.reply.findFirst({
    where: { id: replyId, workspaceSlug },
    select: {
      emailBisonReplyId: true,
      ebSenderEmailId: true,
      leadEmail: true,
    },
  });

  if (!replyRecord?.emailBisonReplyId || !replyRecord.ebSenderEmailId) {
    return NextResponse.json({ error: 'Cannot send: missing sender data' }, { status: 422 });
  }

  const workspace = await prisma.workspace.findUnique({ where: { slug: workspaceSlug } });
  const ebClient = new EmailBisonClient(workspace!.emailBisonToken!);

  const result = await ebClient.sendReply(replyRecord.emailBisonReplyId, {
    message,
    sender_email_id: replyRecord.ebSenderEmailId,
    reply_all: true,
  });

  if (!result.data.success) {
    return NextResponse.json({ error: result.data.message }, { status: 422 });
  }

  // Persist the sent message as a Reply record with direction=outbound
  // so it immediately appears in the thread view
  const sentMsg = result.data.reply!;
  await prisma.reply.create({
    data: {
      workspaceSlug,
      senderEmail: sentMsg.primary_to_email_address,  // outbound: "to" is the lead
      subject: sentMsg.subject,
      bodyText: sentMsg.text_body ?? '',
      htmlBody: sentMsg.html_body,
      receivedAt: new Date(sentMsg.date_received ?? sentMsg.created_at),
      emailBisonReplyId: sentMsg.id,
      emailBisonParentId: sentMsg.parent_id,
      ebSenderEmailId: sentMsg.sender_email_id,
      leadEmail: replyRecord.leadEmail,
      source: 'portal_send',
      direction: 'outbound',
    },
  });

  return NextResponse.json({ success: true });
}
```

### Collapsible AI Suggestion Card

```tsx
// Source: React state pattern — no library needed
'use client';
import { useState } from 'react';
import { ChevronDown, ChevronUp, Sparkles } from 'lucide-react';

export function AISuggestionCard({
  suggestion,
  onUse,
}: {
  suggestion: string;
  onUse: (text: string) => void;

  return (
    <div className="rounded-lg border border-brand/30 bg-brand/5 p-3 mb-3">
      <div className="flex items-center justify-between">
        <div className="flex items-center gap-2 text-sm font-medium text-brand">

        </div>
        <button onClick={() => setExpanded((p) => !p)} className="text-muted-foreground hover:text-foreground">
          {expanded ? <ChevronUp className="h-4 w-4" /> : <ChevronDown className="h-4 w-4" />}
        </button>
      </div>
      {expanded && (
        <div className="mt-2 space-y-2">
          <p className="text-sm text-foreground/80 whitespace-pre-wrap">{suggestion}</p>
          <div className="flex gap-2">
            <button
              onClick={() => onUse(suggestion)}
              className="text-xs px-3 py-1.5 rounded-md bg-brand text-brand-foreground hover:bg-brand/90 font-medium"

              Use this

              Dismiss
            </button>
          </div>
        </div>
      )}
    </div>
  );
}
```

## Open Questions

1. **Does `poll-replies` store `html_body` from EB?**
   - What we know: The webhook handler stores `text_body` only (`reply.html_body` fallback). The poll-replies cron also stores only text body.
   - What's unclear: Whether `html_body` is reliably populated in the EB API response for all reply types.
   - Recommendation: In the migration plan, add `htmlBody: reply.html_body ?? null` to both webhook + poll upsert. If null, fall back to `bodyText` in the conversation view.

2. **Should outbound "Sent" replies from EB be polled into the local DB?**
   - What we know: `getRepliesPage()` returns all reply types including `folder="Sent"`. The current Reply model has no `direction` field — everything is treated as inbound.
   - What's unclear: Whether the poll-replies cron currently ingests Sent folder replies (it likely does since there's no folder filter).
   - Recommendation: Add `direction` field (`inbound` | `outbound`, default `inbound`) to Reply model in the schema migration. Set `direction: reply.folder === 'Sent' ? 'outbound' : 'inbound'` in webhook + poll routes. This solves thread status derivation and prevents duplicate display.

3. **How many replies does a typical workspace have?**
   - What we know: Current DB has ~14,563 people. Reply count unknown but likely hundreds to low thousands per active workspace.
   - What's unclear: Whether capping the thread list query at `take: 200` is sufficient or if we need cursor pagination.
   - Recommendation: Start with `take: 200` sorted by most recent. Implement cursor pagination only if needed (keep it simple per the "pagination or infinite scroll" discretion).

4. **EmailBison workspace token per workspace?**
   - What we know: `EmailBisonClient` is instantiated per token. The Workspace model has `emailBisonToken`.
   - What's unclear: The portal API routes need to get the EB token for the portal user's workspace.
   - Recommendation: `getPortalSession()` returns `workspaceSlug`. Do `prisma.workspace.findUnique({ where: { slug: workspaceSlug }, select: { emailBisonToken: true } })` in the reply send route. Same pattern as existing admin API routes.
