# Phase 54.1: Agent Memory Write-Back - Research

**Researched:** 2026-03-30
**Domain:** Agent post-execution hooks, file I/O, memory persistence
**Confidence:** HIGH

## Summary

Phase 54.1 closes a gap from v7.0: the memory namespace (Phase 47) and seeding script (`scripts/nova-memory.ts`) create per-workspace memory files at `.nova/memory/{slug}/`, and agent rules (all 7 rules files in `.claude/rules/`) define "Memory Write Governance" sections specifying which files each agent may/must-not write to. However, no agent actually writes to these files. The `runAgent()` function in `runner.ts` returns immediately after `generateText()` completes — there is no post-execution hook.

The fix is straightforward: (1) add an optional `onComplete` callback to `AgentConfig` in `types.ts`, (2) invoke it in `runner.ts` after successful execution, (3) create an `appendToMemory()` utility function, and (4) wire insight extraction in each of the 4 specialist agent wrappers (`writer.ts`, `leads.ts`, `campaign.ts`, `research.ts`). The orchestrator does NOT get memory writes — it delegates, not executes.

**Primary recommendation:** Add `onComplete` callback to `AgentConfig`, create `src/lib/agents/memory.ts` with `appendToMemory()`, and wire 4 post-hooks using each agent's output + workspace slug.

## Standard Stack

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| Node.js `fs/promises` | built-in | Append to `.nova/memory/{slug}/*.md` files | Already used in `scripts/nova-memory.ts` for the same files |
| Zod 4 | already installed | Validate insight structure before writing | Already used across all agent types |

### Supporting

No new dependencies required. This is pure file I/O using existing Node.js built-ins.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| File append (`fs/promises`) | Database table for insights | Over-engineering — memory files are simple markdown, read by CLI skills via `cat`. DB would add latency and break the existing read path. |
| Post-hook in `runAgent()` | Agent tool (let the LLM decide when to write) | Unreliable — LLM may forget to call the tool. Post-hook guarantees execution every time. Also avoids adding a file-write tool to agents (security concern). |
| Extract insights from agent output text | Ask model to produce structured insight JSON | Fragile — would require parsing or a second LLM call. Better to extract from the already-parsed typed output. |

### Pattern 1: Post-Execution Hook via AgentConfig

**What:** Add an optional `onComplete` callback to `AgentConfig` that `runAgent()` calls after successful execution, passing the full `AgentRunResult`. This keeps the runner generic and pushes agent-specific logic to the config.

**When to use:** When you need guaranteed post-execution behavior without modifying the core runner for each agent.

**Example:**
```typescript
// In types.ts — extend AgentConfig
export interface AgentConfig {
  // ... existing fields ...
  onComplete?: (result: AgentRunResult, options?: { workspaceSlug?: string }) => Promise<void>;
}

// In runner.ts — after updating audit record, before return
if (config.onComplete) {
  try {
    await config.onComplete({ output, text: result.text, steps, durationMs }, options);
  } catch (hookError) {
    // Log but don't fail the agent run — memory write is best-effort
    console.error(`[Agent ${config.name}] onComplete hook failed:`, hookError);
  }
}
```

### Pattern 2: appendToMemory Utility

**What:** A single function that handles file existence check, path resolution, line counting (max 200 lines per governance rules), and atomic append.

**When to use:** Called from every agent's `onComplete` hook.

**Example:**
```typescript
// In memory.ts
import { appendFile, readFile, access } from "fs/promises";
import { constants } from "fs";
import { join } from "path";

const MEMORY_ROOT = ".nova/memory";
const MAX_LINES = 200;

export async function appendToMemory(
  slug: string,
  file: "campaigns.md" | "feedback.md" | "learnings.md",
  entry: string
): Promise<boolean> {
  const filePath = join(
    process.env.PROJECT_ROOT ?? process.cwd(),
    MEMORY_ROOT,
    slug,
    file
  );

  // Check file exists (don't create — seeding handles that)
  try {
    await access(filePath, constants.F_OK);
  } catch {
    console.warn(`[memory] File not found, skipping: ${filePath}`);
    return false;
  }

  // Check line count to enforce max 200 lines governance
  const content = await readFile(filePath, "utf8");
  const lineCount = content.split("\n").length;
  if (lineCount >= MAX_LINES) {
    console.warn(`[memory] ${filePath} at max lines (${lineCount}), skipping append`);
    return false;
  }

  // Format: [ISO-DATE] — {insight}
  const timestamp = new Date().toISOString();
  const line = `[${timestamp}] — ${entry}\n`;

  await appendFile(filePath, line, "utf8");
  return true;
}
```

### Pattern 3: Per-Agent Insight Extraction

**What:** Each agent's `onComplete` hook extracts relevant insights from its typed output and calls `appendToMemory()` with the appropriate file(s).

**When to use:** In each agent's config definition (writer.ts, leads.ts, campaign.ts, research.ts).

**Key insight extraction rules (from Memory Write Governance in rules files):**

| Agent | May Write To | What To Extract |
|-------|-------------|-----------------|
| Writer | `campaigns.md`, `feedback.md`, `learnings.md` | Strategy used, copy approach, KB references cited |
| Leads | `learnings.md`, `feedback.md` | Source quality, ICP fit observations, filter effectiveness |
| Campaign | `campaigns.md`, `feedback.md`, `learnings.md` | Campaign creation details, approval patterns |
| Research | `learnings.md` | ICP discoveries, website analysis insights |

**Example (writer):**
```typescript
const writerConfig: AgentConfig = {
  name: "writer",
  model: NOVA_MODEL,
  systemPrompt: WRITER_SYSTEM_PROMPT + USER_INPUT_GUARD,
  tools: writerTools,
  maxSteps: 20,
  outputSchema: writerOutputSchema,
  onComplete: async (result, options) => {
    const slug = options?.workspaceSlug;
    if (!slug) return;

    const output = result.output as WriterOutput;

    // Extract what strategy worked and which KB docs were referenced
    const strategy = output.strategy ?? "pvp";
    const refs = output.references?.join(", ") ?? "none";
    const channel = output.channel;

    await appendToMemory(
      slug,
      "campaigns.md",
      `${output.campaignName}: generated ${channel} sequence using ${strategy} strategy. KB refs: ${refs}`
    );
  },
};
```

### Anti-Patterns to Avoid

- **DO NOT give agents a file-write tool**: This was considered and rejected. Agents should not have filesystem write access — the `onComplete` hook runs in trusted code outside the LLM sandbox, extracting insights deterministically from typed output.
- **DO NOT write to `profile.md`**: Every rules file explicitly forbids this. Profile is seed-only, regenerated by `nova-memory.ts`.
- **DO NOT make a second LLM call to "summarize insights"**: Extract from the already-parsed typed output. A second call adds latency, cost, and unreliability.
- **DO NOT block agent return on memory write failure**: Memory is best-effort. Wrap in try/catch, log errors, never throw.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| File existence checking | Custom stat/try-catch | `fs/promises.access()` | Already used in `nova-memory.ts`, consistent pattern |
| Markdown memory format | Custom parser | Simple `appendFile()` with formatted string | Memory files are append-only markdown, no parsing needed |
| Line counting for max-lines | Streaming reader | `readFile().split("\n").length` | Files are capped at 200 lines (~10KB max), full read is fine |

**Key insight:** The memory files are tiny (max 200 lines). There is no need for streaming, database storage, or complex file management. Simple `appendFile` with a line-count guard is the right approach.

## Common Pitfalls

### Pitfall 1: Writing When No Workspace Slug Available

**What goes wrong:** Some agent calls (especially research) may not have a `workspaceSlug`. Attempting to construct a file path with `undefined` slug crashes or writes to wrong location.
**Why it happens:** The orchestrator can call agents without a workspace context (e.g., "analyze this URL" without linking to a workspace).
**How to avoid:** Guard every `onComplete` hook with `if (!slug) return;`. This is already shown in the pattern above.
**Warning signs:** File writes to `.nova/memory/undefined/` directory.

### Pitfall 2: Concurrent Writes to Same File

**What goes wrong:** Two parallel agent runs for the same workspace could race on `appendFile`, potentially interleaving lines.
**Why it happens:** `appendFile` on most OS/filesystem combinations is atomic for small writes, but the read-count-then-append sequence is not atomic.
**How to avoid:** This is acceptable risk for this use case. Memory entries are single lines; even if two appends interleave, both entries land in the file. The max-line check might allow one extra entry past 200 — harmless. Do not add file locking complexity.
**Warning signs:** Line count slightly exceeding 200 (e.g., 201-202 lines). Acceptable.

### Pitfall 3: Memory File Not Yet Seeded

**What goes wrong:** Agent runs before `nova-memory.ts seed` has been executed for a workspace. Memory files don't exist.
**Why it happens:** New workspace hasn't been seeded yet, or seed was run before workspace existed.
**How to avoid:** `appendToMemory()` checks file existence with `access()` and returns `false` if missing. Does not create files — that's the seed script's job.
**Warning signs:** Console warnings `[memory] File not found, skipping`.

### Pitfall 4: PROJECT_ROOT Not Set in Deployed Context

**What goes wrong:** `process.cwd()` fallback may resolve to wrong directory in Vercel or non-CLI contexts.
**Why it happens:** Memory writes are primarily a CLI-mode concern. In API mode (Vercel), the `.nova/memory/` directory doesn't exist on the ephemeral filesystem.
**How to avoid:** `appendToMemory()` should gracefully handle missing directories (the `access()` check covers this). Memory writes only matter in CLI mode where the project root is the working directory. In API mode, the `onComplete` hook will simply log a warning and skip.
**Warning signs:** Warnings in Vercel logs about missing files — expected and harmless.

### Pitfall 5: Overly Verbose Insights

**What goes wrong:** Writing too many entries per run, or entries that are too detailed, quickly fills the 200-line cap.
**Why it happens:** Eager insight extraction from every field in the output.
**How to avoid:** Each `onComplete` should write at most 1-2 entries per run. Entries should be one-line summaries, not multi-line descriptions. Follow the format in the rules: `[ISO-DATE] — {concise insight in one line}`.
**Warning signs:** Memory files hitting 200 lines after only 20-30 agent runs.

## Code Examples

### Complete appendToMemory Implementation

```typescript
// src/lib/agents/memory.ts
import { appendFile, readFile, access } from "fs/promises";
import { constants } from "fs";
import { join } from "path";

const MEMORY_ROOT = ".nova/memory";
const MAX_LINES = 200;

type MemoryFile = "campaigns.md" | "feedback.md" | "learnings.md";

/**
 * Append an insight entry to a workspace memory file.
 *
 * - Validates the file exists (does NOT create — seed script handles that)
 * - Enforces 200-line max per governance rules
 * - Formats entry with ISO timestamp
 * - Best-effort: logs warnings on failure, never throws
 *
 * @returns true if appended, false if skipped (file missing, at capacity, etc.)
 */
export async function appendToMemory(
  slug: string,
  file: MemoryFile,
  entry: string,
): Promise<boolean> {
  try {
    const projectRoot = process.env.PROJECT_ROOT ?? process.cwd();
    const filePath = join(projectRoot, MEMORY_ROOT, slug, file);

    // File must exist (seeded by nova-memory.ts)
    try {
      await access(filePath, constants.F_OK);
    } catch {
      console.warn(`[memory] File not found, skipping: ${filePath}`);
      return false;
    }

    // Enforce max line count
    const content = await readFile(filePath, "utf8");
    const lineCount = content.split("\n").length;
    if (lineCount >= MAX_LINES) {
      console.warn(`[memory] ${slug}/${file} at max lines (${lineCount}), skipping`);
      return false;
    }

    // Append with ISO timestamp
    const timestamp = new Date().toISOString();
    const line = `[${timestamp}] — ${entry}\n`;
    await appendFile(filePath, line, "utf8");
    return true;
  } catch (error) {
    console.error(`[memory] Failed to append to ${slug}/${file}:`, error);
    return false;
  }
}
```

## Open Questions

1. **Global insights file write-back**
   - What we know: `intelligence-rules.md` governs writes to `.nova/memory/global-insights.md` (intelligence agent only). The intelligence agent is not one of the 4 specialists wired in `runner.ts` — it's a CLI-only skill.
   - What's unclear: Should this phase also wire global-insights write-back, or defer to a future phase?
   - Recommendation: Defer. The intelligence agent runs via CLI skill, not via `runAgent()`. Global insights writes would need a different mechanism (CLI post-hook or explicit tool). Keep this phase focused on the 4 specialist agents that use `runAgent()`.

2. **Deliverability and onboarding agent write-back**
   - What we know: `deliverability-rules.md` and `onboarding-rules.md` also define Memory Write Governance sections.
   - What's unclear: These agents don't appear to have `runXxxAgent()` functions in `src/lib/agents/` — they may be CLI-only skills.
   - Recommendation: Confirm during planning. If they use `runAgent()`, wire them. If CLI-only, defer.

3. **Should insight extraction be smarter (LLM-based)?**
   - What we know: Current plan extracts insights deterministically from typed output fields.
   - What's unclear: Whether deterministic extraction produces useful insights vs. an LLM summarization step.
   - Recommendation: Start deterministic. The typed output already contains structured data (strategy, channel, references, action, summary). An LLM call adds ~2s latency and token cost for marginal benefit. Can always add later if insights feel too mechanical.
