Create `trigger/process-reply.ts` with the following structure:

**Payload type** — define `ProcessReplyPayload` interface at the top:
```typescript
interface ProcessReplyPayload {
  workspaceSlug: string;
  ebReplyId: string | number;
  eventType: string;
  leadEmail: string;
  leadName: string | null;
  senderEmail: string | null;
  subject: string | null;
  textBody: string | null;
  interested: boolean;
  campaignId: string | null;
  webhookEventId: string;
  replyFromEmail: string;
  replyFromName: string | null;
  replyBodyText: string;
  replyHtmlBody: string | null;
  replyReceivedAt: string;        // ISO string (dates don't serialize)
  replyParentId: string | null;
  replySenderEmailId: string | null;
  direction: "inbound" | "outbound";
  sequenceStep: number | null;
}
```

**Task config:**
- `id: "process-reply"`
- `queue: anthropicQueue` (from `./queues`)
- `maxDuration: 120` (2 min — classification + notify + AI suggestion)
- `retry: { maxAttempts: 3, factor: 2, minTimeoutInMs: 1000, maxTimeoutInMs: 10_000 }`

**PrismaClient** — instantiate `const prisma = new PrismaClient()` at module scope (same as smoke-test.ts).

**`run()` implementation** — four sequential steps:

1. **Upsert Reply record** — replicate the exact upsert logic from the current webhook handler (lines 392-428 in route.ts). Look up the Outsignal campaign by `emailBisonCampaignId` to get `outsignalCampaignId`, `outsignalCampaignName`, `outboundSubject`, `outboundBody`. Look up `personId` by `replyFromEmail`. Upsert by `emailBisonReplyId`. Use the same field mapping.

2. **Classify** — call `classifyReply()` (import from `@/lib/classification/classify-reply`). On success, update the Reply record with `intent`, `sentiment`, `objectionSubtype`, `classificationSummary`, `classifiedAt`. On failure, log error — the retry-classification cron picks it up.

3. **Notify** — call `notifyReply()` (import from `@/lib/notifications`) passing `replyId` from step 1. The `notifiedAt` guard inside `notifyReply()` prevents duplicate notifications on task retries. Pass classification info in the notification: if classification succeeded, include intent/sentiment info in the `bodyPreview` field by appending ` [Intent: ${intent}, Sentiment: ${sentiment}]` to the preview text.

4. **Generate AI suggestion** — replicate the `generateReplySuggestion` logic from the webhook (lines 69-96 in route.ts). Only run for LEAD_REPLIED and LEAD_INTERESTED events with non-empty textBody. On success:
   - Update Reply record with `aiSuggestedReply`
   - Post follow-up Slack message to workspace channel + replies channel (same logic as current webhook lines 494-515)
   Use `import { postMessage } from "@/lib/slack"` and `import { generateText } from "ai"` and `import { anthropic } from "@ai-sdk/anthropic"`.

**Important:**
- These lib imports (`classifyReply`, `notifyReply`, `postMessage`) use the singleton `prisma` from `@/lib/db` internally. That's fine — Trigger.dev's Prisma extension handles both the singleton and module-scope instances. The module-scope `prisma` in this task file is for DIRECT queries (upsert, update). The libs use their own.
- Export the task as a named export: `export const processReply = task({ ... })`.
- Export the `ProcessReplyPayload` type for use with `import type` in the webhook handler.
