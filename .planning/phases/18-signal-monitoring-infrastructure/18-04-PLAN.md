Export three functions:

`checkWorkspaceCap(workspaceSlug: string, capUsd: number): Promise<boolean>` — returns true if cap hit:
- Get today's date as "YYYY-MM-DD" UTC
- findUnique on SignalDailyCost where `{ workspaceSlug_date: { workspaceSlug, date } }`
- Return `(record?.totalUsd ?? 0) >= capUsd`

`incrementWorkspaceSpend(workspaceSlug: string, costUsd: number, signalType: string): Promise<void>`:
- Upsert SignalDailyCost: increment `totalUsd` by `costUsd`
- Also update `breakdown` JSON: parse existing breakdown (or {}), increment the key for signalType, JSON.stringify back

`alertBudgetCapHit(workspaceSlug: string, workspaceName: string, capUsd: number, spentUsd: number): Promise<void>`:
- Uses `@slack/web-api` WebClient with `process.env.SLACK_BOT_TOKEN`
- Sends message to `process.env.ADMIN_SLACK_CHANNEL_ID` (or the workspace's slackChannelId if available)
- Message text: `[{workspaceName}] Signal daily budget cap reached`
- Block kit: header + section with workspace name, daily cap, spent amount, "Signal processing paused until midnight UTC"
- If SLACK_BOT_TOKEN not set, log warning and skip (don't crash)

**worker-signals/src/cycle.ts** — The main orchestration loop:

Export `runCycle(): Promise<void>`:

```
1. Run cleanup: expireOldSignals() + cleanupOldSeenUrls() — log counts
2. Load workspace configs: loadWorkspaceConfigs()
   - If no workspaces have signals enabled, log and return early
3. Build domain-workspace map: buildDomainWorkspaceMap(configs)
4. Process PredictLeads signals (domain-based watchlist):
   For each unique domain in the map:
     a. Fetch all enabled signal types for this domain (job openings, financing, news, tech)
        - Only call adapters that at least one workspace has enabled for this domain
        - Calculate sinceDate = 7 hours ago (slightly more than 6h interval to avoid gaps)
     b. For each workspace that watches this domain:
        - Check budget cap: if checkWorkspaceCap returns true, skip + alertBudgetCapHit (once per cycle per workspace)
        - Filter signals to only the types this workspace has enabled
        - writeSignalEvents(filteredSignals, workspaceSlug)
        - incrementWorkspaceSpend(workspaceSlug, costUsd, signalType)
     c. Hiring spike detection:
        - If job openings adapter returned totalJobCount > 10, create a "hiring_spike" SignalEvent for each relevant workspace
5. Process Serper social listening:
   For each workspace with "social_mention" in enabledTypes AND competitors.length > 0:
     a. Check budget cap (same as above)
     b. searchCompetitorMentions(workspace.competitors)
     c. writeSignalEvents(socialSignals, workspace.slug)
     d. incrementWorkspaceSpend for each Serper call
6. Log cycle summary: total signals written, workspaces processed, budget caps hit
```

Key design decisions per CONTEXT.md locked decisions:
- Serper runs AFTER PredictLeads (single worker process, sequential)
- Worker deduplicates domains across workspaces, polls once, fans out
- When cap hit mid-cycle, skip remaining items for that workspace
- Domain processing order: randomize (shuffle) to ensure fair coverage when budget caps hit mid-cycle (per Claude's discretion in CONTEXT.md)
- Error handling: try/catch per domain — log error and continue to next domain. One failed domain does not abort the cycle. For Serper, try/catch per workspace.

**worker-signals/src/index.ts** — Update entry point to wire runCycle:

Replace the stub from Plan 02:
```typescript
import { runCycle } from "./cycle.js";
import { prisma } from "./db.js";

async function main() {
  console.log(`[SignalWorker] Starting cycle at ${new Date().toISOString()}`);
  try {
    await runCycle();
    console.log(`[SignalWorker] Cycle complete at ${new Date().toISOString()}`);
  } catch (error) {
    console.error("[SignalWorker] Fatal error:", error);
    process.exitCode = 1;
  } finally {
    await prisma.$disconnect();
    process.exit(process.exitCode ?? 0);
  }
}

main();
```

The `finally` block ensures Prisma disconnects even on error (preventing Railway from hanging). `process.exitCode` pattern ensures correct exit code on fatal errors.
