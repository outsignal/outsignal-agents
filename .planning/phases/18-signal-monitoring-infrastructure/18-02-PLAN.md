**Dockerfile** — Lightweight (no browser deps like the LinkedIn worker):
```dockerfile
FROM node:22-slim

WORKDIR /app

# Copy Prisma schema from main project (needed for prisma generate)
COPY prisma ./prisma

COPY worker-signals/package.json worker-signals/package-lock.json* ./
RUN npm install

COPY worker-signals/tsconfig.json ./
COPY worker-signals/src ./src
RUN npx prisma generate
RUN npx tsc
RUN npm prune --production

CMD ["node", "dist/index.js"]
```

IMPORTANT: The Dockerfile copies `prisma/` from the repo root because @prisma/client needs the schema for `prisma generate`. This is a monorepo pattern — the Dockerfile is at `worker-signals/Dockerfile` but the Railway build context must be the repo root. The railway.toml handles this with `dockerfilePath`.

**railway.toml:**
```toml
[build]
builder = "dockerfile"
dockerfilePath = "./worker-signals/Dockerfile"

[deploy]
cronSchedule = "0 */6 * * *"
```

The `cronSchedule` makes Railway run the worker every 6 hours at :00 UTC. No `restartPolicyType` — cron services exit after each run.

**worker-signals/src/predictleads/types.ts** — Zod schemas for all 5 PredictLeads signal type responses. These are MEDIUM confidence based on API documentation structure; executor should adjust field names if live API returns different shapes.

Define Zod v3 schemas:
- `JobOpeningSchema` — fields: id (string), title (string), url (string/optional), first_seen_at (string, ISO date), location (string/optional), seniority (string/optional), department (string/optional)
- `FinancingEventSchema` — fields: id (string), funding_type (string, e.g. "seed", "series_a"), amount (number/optional), currency (string/optional), first_seen_at (string), investors (array of strings/optional), source_url (string/optional)
- `NewsEventSchema` — fields: id (string), title (string), category (string/optional), source_url (string/optional), first_seen_at (string), summary (string/optional)
- `TechnologyDetectionSchema` — fields: id (string), technology_name (string), category (string/optional), first_seen_at (string), is_new (boolean/optional)

Each schema uses `z.object({...}).passthrough()` to allow extra fields from the API without failing validation. Export both the Zod schemas and inferred TypeScript types (`z.infer<typeof ...>`).

Also define:
- `PredictLeadsListResponse` — generic wrapper: `{ data: T[], pagination?: { total: number, page: number, per_page: number } }`
- Export type `SignalType = "job_change" | "funding" | "hiring_spike" | "tech_adoption" | "news" | "social_mention"`

**worker-signals/src/predictleads/client.ts** — Low-level HTTP client:

```typescript
const BASE_URL = "https://predictleads.com/api/v3";
const REQUEST_TIMEOUT_MS = 15_000;
const MAX_RETRIES = 3;
const RETRY_BACKOFF_MS = [1000, 3000, 9000]; // exponential
```

Export `predictLeadsGet(path: string): Promise<unknown>`:
- Reads `PREDICTLEADS_API_KEY` and `PREDICTLEADS_API_TOKEN` from env (throw if missing)
- Sends GET to `${BASE_URL}${path}` with headers `X-Api-Key` and `X-Api-Token`
- AbortController timeout at 15s
- On 429: retry with exponential backoff (1s, 3s, 9s), log each retry
- On 4xx/5xx non-429: throw with status code and path
- On success: return `response.json()` result
- On timeout/network error after max retries: throw

Also export `predictLeadsCostPerCall(): number` — returns `0.04` (the $0.04/call rate for 101-5k tier per CONTEXT.md pricing).
