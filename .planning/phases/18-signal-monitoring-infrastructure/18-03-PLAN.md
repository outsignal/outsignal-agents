Each adapter file follows the same pattern:
1. Import `predictLeadsGet` from `./client.js` and the relevant Zod schema from `./types.js`
2. Export a single async function that takes `(domain: string, sinceDate?: string)` and returns `{ signals: SignalInput[], costUsd: number, rawResponse: unknown }`
3. Call PredictLeads API with the domain and optional date filter
4. Parse response using the Zod schema (`.safeParse` — log and skip invalid records, don't throw)
5. Convert each valid result to a `SignalInput` object (defined below)

**Common return type** (define in a new `worker-signals/src/types.ts` or inline):
```typescript
export interface SignalInput {
  signalType: SignalType;
  source: "predictleads" | "serper";
  externalId: string | null;
  companyDomain: string;
  companyName?: string;
  title?: string;
  summary?: string;
  confidence?: number;
  sourceUrl?: string;
  rawResponse: string; // JSON.stringify of raw API item
  metadata?: string; // JSON.stringify of extra structured data
}
```

**job-openings.ts** — `fetchJobOpenings(domain, sinceDate?)`:
- GET `/companies/${domain}/job_openings` (append `?first_seen_at_from=${sinceDate}` if provided)
- Each job opening becomes a signal with `signalType: "job_change"`, `externalId` from the API record id
- `title` = job title, `metadata` = JSON with location, seniority, department
- Also returns the raw job count for hiring spike detection (see below)

**Hiring spike detection** — The same `fetchJobOpenings` function returns `{ signals, costUsd, rawResponse, totalJobCount }`. The cycle orchestrator (Plan 04) compares `totalJobCount` against a threshold to decide whether to also create a `"hiring_spike"` SignalEvent. Threshold: >10 new job openings in a single fetch = hiring spike. This avoids needing a separate API call — hiring spikes are derived from job openings volume.

**financing.ts** — `fetchFinancingEvents(domain, sinceDate?)`:
- GET `/companies/${domain}/financing_events`
- `signalType: "funding"`, `title` = funding type + amount, `metadata` = JSON with funding_type, amount, currency, investors

**news.ts** — `fetchNewsEvents(domain, sinceDate?)`:
- GET `/companies/${domain}/news_events`
- `signalType: "news"`, `title` = news title, `summary` = news summary, `sourceUrl` = source URL

**technology.ts** — `fetchTechnologyDetections(domain, sinceDate?)`:
- GET `/companies/${domain}/technology_detections`
- `signalType: "tech_adoption"`, `title` = "Adopted {technology_name}", `metadata` = JSON with technology_name, category, is_new

All adapters set `costUsd` to `predictLeadsCostPerCall()` (one API call per adapter invocation). Error handling: if PredictLeads returns 404 (company not found), return empty signals array with costUsd=0 (don't charge for not-found domains). For other errors, let them propagate up.

import { PrismaClient } from "@prisma/client";

**`writeSignalEvents(signals: SignalInput[], workspaceSlug: string): Promise<number>`**
- Takes array of SignalInput and workspace slug
- For each signal, upserts to SignalEvent table using `@@unique([source, externalId])`:
  - If `externalId` is null, use `create` (no upsert possible — social mentions have no external ID)
  - If `externalId` is present, use `upsert` with `where: { source_externalId: { source, externalId } }`
  - Set `expiresAt` to `new Date(Date.now() + 90 * 24 * 60 * 60 * 1000)` (90 days from now)
- Returns count of signals written (created or updated)
- After writing, call `checkAndFlagHighIntent` for each unique companyDomain in the batch

**`checkAndFlagHighIntent(companyDomain: string, workspaceSlug: string): Promise<void>`**
- Query SignalEvent for this company+workspace where `status = "active"` and `detectedAt >= 30 days ago`
- Count distinct `signalType` values
- If distinctTypes >= 2, set `isHighIntent = true` on ALL active signals for this company+workspace
- If distinctTypes < 2, set `isHighIntent = false` (signal may have expired, removing the stacking)

**`expireOldSignals(): Promise<number>`**
- `updateMany` where `status = "active"` AND `expiresAt < now()`
- Set `status = "expired"`
- Returns count of expired signals
- This runs at the start of each cycle (called from cycle.ts in Plan 04)

Also export `disconnectPrisma(): Promise<void>` which calls `prisma.$disconnect()` — needed before `process.exit(0)` to cleanly close the DB connection pool.
