**`compileTemplate(template: string, context: Record<string, unknown>): string`**
- Import Handlebars from 'handlebars'
- Compile with `{ noEscape: true }` (LinkedIn messages are plain text, no HTML escaping needed)
- Return compiled template with context applied
- Handle errors gracefully: if compilation fails, return the raw template string with a console.warn

**`buildTemplateContext(person: { firstName?: string | null; lastName?: string | null; company?: string | null; jobTitle?: string | null; linkedinUrl?: string | null }, emailContext?: { stepRef?: string; subject?: string; opened?: boolean; clicked?: boolean }): Record<string, unknown>`**
- Maps person fields to template variables: firstName, lastName, companyName, jobTitle, linkedinUrl
- Maps email context: emailStepRef, emailSubject, emailOpened, emailClicked
- Defaults all nullable values to empty string for safe template rendering

**`evaluateSequenceRules(params: { workspaceSlug: string; campaignName: string; triggerEvent: 'email_sent' | 'connection_accepted'; triggerStepRef?: string; personId: string; person: { firstName?, lastName?, company?, jobTitle?, linkedinUrl?, email: string }; emailContext?: { stepRef?: string; subject?: string; opened?: boolean; clicked?: boolean }; senderEmail?: string }): Promise<Array<{ actionType: string; messageBody: string | null; delayMinutes: number; sequenceStepRef: string }>>`**

Logic:
1. Query CampaignSequenceRules matching workspaceSlug + campaignName + triggerEvent
2. If triggerStepRef provided: filter rules where triggerStepRef matches
3. For each matching rule:
   a. If rule.requireConnected: check LinkedInConnection status — skip if not 'connected'
   b. Build template context from person + emailContext
   c. If rule.messageTemplate: compile with Handlebars → messageBody
   d. Return action descriptor: { actionType: rule.actionType, messageBody, delayMinutes: rule.delayMinutes, sequenceStepRef: `rule_${rule.id}` }
4. Return array of action descriptors (caller enqueues them)

**`createSequenceRulesForCampaign(params: { workspaceSlug: string; campaignName: string; linkedinSequence: Array<{ position: number; type: string; body?: string; delayHours?: number; triggerEvent?: string; triggerStepRef?: string; requireConnected?: boolean }> }): Promise<void>`**

Creates CampaignSequenceRule records from the campaign's linkedinSequence during deploy.
- For each step in linkedinSequence:
  - triggerEvent: step.triggerEvent ?? (position === 1 ? 'delay_after_previous' : 'email_sent')
  - triggerStepRef: step.triggerStepRef ?? null
  - actionType: step.type (connect, message, profile_view)
  - messageTemplate: step.body ?? null
  - delayMinutes: (step.delayHours ?? 0) * 60
  - requireConnected: step.requireConnected ?? (step.type === 'message')
  - position: step.position
- Use prisma.campaignSequenceRule.createMany for batch insert

Create src/lib/linkedin/connection-poller.ts — connection accept detection and timeout handling.

**Constants:**
- `CONNECTION_TIMEOUT_DAYS = 14` — after this many days with no accept, withdraw
- `WITHDRAWAL_COOLDOWN_HOURS = 48` — wait this long after withdraw before retrying
- `MAX_RETRY_ATTEMPTS = 1` — only retry once after timeout

**Export `pollConnectionAccepts(workspaceSlug: string): Promise<{ checked: number; accepted: number; timedOut: number; failed: number }>`**

Logic:
1. Find all pending connections for the workspace:
   ```typescript
   const pendingConnections = await prisma.linkedInConnection.findMany({
     where: {
       status: 'pending',
       sender: { workspaceSlug, status: 'active', healthStatus: 'healthy' }
     },
     include: {
       sender: true,
     }
   });
   ```

2. For each pending connection:
   a. **Check timeout first:** If requestSentAt + CONNECTION_TIMEOUT_DAYS < now:
      - Check if there's already a failed retry attempt (LinkedInAction with sequenceStepRef='connection_retry' for this person):
        - If yes: mark connection as 'failed', skip remaining LinkedIn steps:
          ```typescript
          await prisma.linkedInAction.updateMany({
            where: { personId: conn.personId, workspaceSlug, status: 'pending' },
            data: { status: 'cancelled' }
          });
          ```
        - If no: check if cooldown period has passed since requestSentAt + TIMEOUT:
          - If cooldown passed: enqueue a retry connection request with sequenceStepRef='connection_retry'
          - If cooldown not passed: skip (will be processed next poll cycle)
      - Continue to next connection (don't check status via API — it's timed out)

   b. **Not timed out — return action descriptor for worker to check:**
      This function returns the list of connections to check. The actual VoyagerClient.checkConnectionStatus() call happens in the worker (Plan 03 handles wiring).

      For now, this function queries and returns the data. The worker calls a separate function to process results.

**Export `processConnectionCheckResult(connectionId: string, newStatus: 'connected' | 'none' | 'failed'): Promise<void>`**

Logic:
1. Load the connection record
2. If newStatus === 'connected':
   a. Update LinkedInConnection: status → 'connected', connectedAt = now
   b. Look up CampaignSequenceRules with triggerEvent = 'connection_accepted' matching the campaign
   c. For each rule: call evaluateSequenceRules from sequencing.ts
   d. For each returned action: assignSenderForPerson, then enqueueAction with scheduledFor = now + delayMinutes
   e. If original sender is flagged (healthStatus !== 'healthy'): try to assign a different healthy sender. If none available, log warning and hold (action stays pending, will be picked up when a sender recovers)

3. If newStatus === 'failed' (declined/withdrawn by the other party):
   a. Update LinkedInConnection: status → 'failed'
   b. Cancel all remaining pending LinkedIn actions for this person in the workspace

**Export `getConnectionsToCheck(workspaceSlug: string): Promise<Array<{ connectionId: string; senderId: string; personId: string; personLinkedinUrl: string }>>`**

Returns connections that need to be checked via VoyagerClient. Used by the worker to batch connection status checks.
- Filters: status === 'pending', not timed out, sender is active+healthy
- Includes person.linkedinUrl for the VoyagerClient call
