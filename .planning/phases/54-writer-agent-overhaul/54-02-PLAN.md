<objective>
Add campaign-holistic awareness and KB citation enforcement to the writer agent. The writer must see all existing campaign steps before generating new ones, track used angles/CTAs to prevent duplication, and cite specific KB principles applied in each step.

Purpose: Currently the writer can generate copy in isolation without seeing the full campaign context, leading to repeated CTAs and angles across steps. KB consultation produces references but no evidence of application. This plan makes the writer campaign-aware and KB-accountable.

Output: Writer loads full campaign context before generating, avoids duplicate angles/CTAs, and cites applied KB principles in notes.

<feature>
  <name>Campaign-Holistic Awareness + KB Citation Enforcement + Cross-Step Dedup</name>
  <files>src/lib/agents/writer.ts, src/lib/agents/types.ts, .claude/rules/writer-rules.md</files>
  <behavior>
    **1. Campaign-Holistic Awareness (COPY-03):**
    - When campaignId is present in the writer input, the system prompt mandates:
      a) Call getCampaignContext FIRST to load all existing sequences (email + LinkedIn)
      b) Before generating, enumerate all existing angles (the core pitch/hook of each step) and CTAs (the closing question)
      c) New steps must use DIFFERENT angles and CTAs from existing ones
      d) If generating step 3 of a 3-step sequence, load steps 1 and 2 and ensure step 3 has a fresh angle
    - This is a system prompt behavioral change — the tools already exist
    - The validateCopy tool adds a cross-step CTA dedup check: if two steps end with identical CTA phrasing (exact match after lowercasing and trimming), return a soft violation

    **2. KB Citation Enforcement (COPY-06):**
    - System prompt mandates: after KB search returns results, the writer must apply specific principles and document them
    - Every step's `notes` field must contain: "Applied: [specific principle name] from [KB doc title] — [one-line explanation of how it shaped this step]"
    - If KB search returns zero results for the primary query, note in reviewNotes: "No KB docs found for [strategy]+[vertical]. Using general best practices."
    - The `references` field in WriterOutput continues to list KB doc titles, but the per-step `notes` field is the primary traceability mechanism
    - The validateCopy tool adds a soft check: if any step's notes field does not contain "Applied:" and the writer output has non-empty references, warn about missing citation

    **3. Cross-Step CTA Dedup in validateCopy:**
    - After running per-step structural checks, the validateCopy tool does a cross-step check:
      a) Extract the last sentence (CTA) from each step's body
      b) Compare all CTAs pairwise (case-insensitive, trimmed)
      c) If any two steps share the same CTA text, return a soft violation: "Steps {a} and {b} share identical CTA: '{cta}' — vary the closing question"
    - This catches exact duplicates. Semantic similarity is Phase 55 (Validator Agent).

    **4. Escalation with Review Notes:**
    - System prompt + Self-Review Protocol (from 54-01) already mandate max 2 retries
    - This plan adds to the system prompt: when saving with violations, prefix the step's notes with "[REVIEW NEEDED] Remaining violation: {description}. "
    - Admin sees this in the campaign approval flow — the notes field is visible in the review UI
  </behavior>
  <implementation>
    **Step 1: Update writer-rules.md with Campaign-Holistic Awareness section**
    - Add new section "## Campaign-Holistic Awareness (MANDATORY when campaignId provided)" before the Copy Strategies section:
      ```
      When a campaignId is provided in your task:
      1. MUST call getCampaignContext FIRST before any other tool
      2. If the campaign has existing email or LinkedIn sequences, read every step
      3. Build an internal tracking list:
         - Taken angles: the core hook/pitch of each existing step (one line per step)
         - Taken CTAs: the closing question of each existing step
      4. When generating new steps, pick DIFFERENT angles and CTAs from your tracking list
      5. If generating a replacement for a specific step (stepNumber provided), you may reuse that step's angle but must not duplicate other steps' angles
      6. Angle deduplication is within the current campaign only — cross-campaign angle variety is encouraged
      ```

    **Step 2: Update writer-rules.md with KB Citation Requirements section**
    - Add new section "## KB Citation Requirements (MANDATORY)" after Campaign-Holistic Awareness:
      ```
      After running KB search (searchKnowledgeBase), you MUST:
      1. If results returned: identify the most relevant principle and apply it to your copy
      2. In each step's `notes` field, include: "Applied: [principle name] from [KB doc title] — [how it shaped this step]"
      3. If KB search returns 0 results for the primary strategy+industry query, note in reviewNotes: "No KB docs found for [query]. Using general best practices."
      4. The `references` array should list all KB doc titles consulted
      5. Every strategy (PVP, Creative Ideas, One-liner, Custom) must trace its core angle to a KB doc, case study, or differentiator — not just Creative Ideas
      ```

    **Step 3: Enhance validateCopy tool for cross-step CTA dedup**
    - After per-step validation, extract last sentence from each step's body
    - Compare pairwise (case-insensitive trim)
    - Add soft violation if duplicates found
    - Add soft check for missing "Applied:" in notes when references exist

    **Step 4: Update WRITER_SYSTEM_PROMPT in writer.ts**
    - The rules are loaded via loadRules() so the writer-rules.md changes auto-load
    - Add to the system prompt constant (not in rules file): a brief instruction reinforcing the mandate:
      "CRITICAL WORKFLOW: When campaignId is provided, you MUST call getCampaignContext before generating. You MUST call validateCopy before saving. See your rules for the full Self-Review Protocol, Campaign-Holistic Awareness, and KB Citation Requirements."

    **Step 5: Update WriterOutput type comments in types.ts**
    - Add JSDoc to `references` field: "KB doc titles cited. Per-step application details go in each step's `notes` field as 'Applied: [principle] from [doc]'."
    - Add JSDoc to EmailStep.notes and LinkedInStep.notes: "Include 'Applied: [principle] from [KB doc]' citation. Prefix with '[REVIEW NEEDED]' if violations persist after 2 rewrite attempts."
  </implementation>
