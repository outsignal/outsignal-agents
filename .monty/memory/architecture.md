[2026-04-16T19:30:00Z] -- Observation (diagnostic, not a pattern — a warning): LinkedIn rendering path is STRUCTURALLY DIVERGENT from EmailBison rendering path, and no code enforces parity. Writer emits canonical `{FIRSTNAME}`/`{COMPANYNAME}` tokens (single-curly UPPERCASE — enforced by src/lib/copy-quality.ts BANNED_PATTERNS). Email path: `transformVariablesForEB` (src/lib/emailbison/variable-transform.ts) maps these to EB's vendor-shaped `{FIRST_NAME}`/`{COMPANY}` tokens at the wire boundary. Company normalisation: `normalizeCompanyName` (src/lib/emailbison/company-normaliser.ts) applied at email-adapter.ts Step 4. LinkedIn path: `compileTemplate` (src/lib/linkedin/sequencing.ts) uses Handlebars expecting `{{firstName}}`/`{{companyName}}` (double-curly camelCase). There is NO transformer on the LinkedIn side — writer tokens pass through Handlebars UNTOUCHED and hit LinkedIn verbatim. Company values pass through unnormalised (bypassing BL-103's work entirely). Sender name placeholders (`{SENDER_FIRST_NAME}`) only exist on the email side — LinkedIn's buildTemplateContext has no sender fields. For any future multi-channel feature touching copy-quality, the rule is: copy-quality's variable-format invariants must be channel-aware, OR every channel must ship a transformer that converts canonical writer tokens to its native render syntax. A single writer + two divergent render paths without an adapter layer is a latent incident generator — variable bugs become undetectable in review because the banned-pattern check validates against the format of channel A while channel B silently ships raw template text. BlankTag diagnostic (2026-04-16) confirmed all 4 BlankTag LinkedIn campaigns ship `{FIRSTNAME}` literally to prospects today; the email side works because transformVariablesForEB exists, the LinkedIn side doesn't because no equivalent transformer was ever built. Future work recommendation (not done yet): add `transformVariablesForLinkedIn` with the same shape as transformVariablesForEB (canonical-token input → channel-native token output, idempotent, unmapped-token warn), call it at the LinkedIn render boundary in createSequenceRulesForCampaign OR compileTemplate. Apply `normalizeCompanyName` at buildTemplateContext (or move the normaliser to `src/lib/common/` to make cross-channel use cleaner). Add sender context fields to buildTemplateContext (requires sender resolution at render time — connection-poller.ts already knows the sender via conn.sender.id).

[2026-04-16T10:25:00Z] -- Pattern: shared narrowing helper for cross-class error semantics. When a single domain throws multiple Error subclasses that share a semantic (e.g. "record not found" expressed as both HTTP-404 and 200-with-empty-data), extract an `isXxx(e: unknown): boolean` helper that tests each class with instanceof + property checks. Callers use `if (isXxx(err)) { /* domain branch */ } else { throw err; }` rather than narrowing to one class and missing the other. First use: BL-078 `isNotFoundError` in `src/lib/emailbison/errors.ts`, covering `EmailBisonApiError` (HTTP 404, has `.status + .isRecordNotFound` getter) AND `EmailBisonError` (200-with-empty-data during EB's async DELETE queue window, has `.code + .statusCode`). Rules: (1) return plain `boolean`, not a type predicate — predicate narrowing would falsely imply the error is one of N specific classes and block legitimate access to class-specific properties (e.g. `.body` for logging) in the branch that used `instanceof` directly. (2) Never match on duck-typed plain objects (`{status: 404}`) — only match real instanceof. (3) Keep the helper file free of downstream imports: if class A lives in `client.ts` and class B in `types.ts`, the helper lives in its own `errors.ts` that imports from both, so `client.ts` can't import the helper back without refactoring class locations (circular dep). Internal methods on the class that throws can't use their own helper — callers external to the domain can. (4) Cover non-Error inputs (null/undefined/string/plain-object/symbol/number) explicitly in unit tests — `instanceof` returns false for all of them but the test matrix makes the contract explicit to future readers. Reuse pattern for any cross-class domain semantic (auth errors, rate-limit errors, permission-denied errors that get thrown as different classes by different transport layers).

[2026-04-16T14:00:00Z] -- Pattern: sibling-row retry anchor for rollback-then-retry flows. When a terminal-failure compensating transaction MUST clear state fields (e.g. a status guard, a foreign-key pointer) that a subsequent task-level retry ALSO needs to reuse, anchor the retry-recovery on a SIBLING row that the rollback intentionally leaves untouched. The retry-entry code reads the sibling row and restores the rolled-back state before the guard fires. First use: BL-076 `executeDeploy` at src/lib/campaigns/deploy.ts — Bundle B's rollback clears `Campaign.status`, `Campaign.emailBisonCampaignId`, `Campaign.deployedAt`; Bundle C's retry-entry reads `CampaignDeploy.emailBisonCampaignId` (a sibling row Bundle B does NOT clear) to restore Campaign state atomically. Rules: (1) The sibling-row write MUST precede any non-idempotent external call that could leave orphan side-effects; in email-adapter Step 2 this meant reordering `prisma.campaignDeploy.update` BEFORE `prisma.campaign.update` so the anchor is durable even if the process dies between the two writes. (2) The restore write uses the same race-safe optimistic-update pattern as the forward transition (`updateMany({where:{id,status:'approved',emailBisonCampaignId:null},...})` guards against double-restore on a retry whose rollback already succeeded the first time). (3) Wrap restore in a $transaction so a concurrent observer never sees Campaign restored while CampaignDeploy still shows the stale terminal state. (4) Defensive-guard at entry — refuse to proceed on any unexpected combination (e.g. status='failed' AND anchor=null means prior attempt died before Step 2 persisted; proceeding could duplicate). (5) Remove client-layer auto-retry on non-idempotent external POSTs (e.g. `withRetry(createCampaign)`) — transient failures must route through the task-level retry that uses the idempotent reuse path, not duplicate the external resource. Pair this with the earlier findFirst-inside-tx pattern (12:45:00Z): Bundle B's rollback gate handles concurrent external initiates (Scenario A: two CampaignDeploy rows); Bundle C's anchor-restore handles task-level retries (Scenario B: one CampaignDeploy row re-invoked). Orthogonal, composable.

[2026-04-16T12:45:00Z] -- Pattern: findFirst-inside-tx for retry-aware atomic rollback. Any compensating transaction whose behaviour depends on the absence of a sibling row MUST run the `findFirst` inside the same `$transaction` as the write, not outside it. Reading the sibling status on the outer client and then opening a tx leaves a race hole where a second actor can enqueue the missing sibling between the read and the write. First use: BL-075 `executeDeploy` atomic Campaign.status rollback in `src/lib/campaigns/deploy.ts` — the outer catch opens a `$transaction`, inside which it (1) writes `CampaignDeploy.status='failed'` guarded by `currentDeploy.status==='running'`, (2) runs `tx.campaignDeploy.findFirst({where:{campaignId, status:{in:['pending','running']}, id:{not:deployId}}})` to check for an inflight retry sibling, (3) if absent, snapshots `Campaign.emailBisonCampaignId`, runs `tx.campaign.updateMany({where:{id,status:'deployed'},data:{status:'approved',emailBisonCampaignId:null,deployedAt:null}})` (race-safe optimistic-update pattern from `deploy-campaign.ts:130-133`), and (4) writes a `tx.auditLog.create` row. The audit is silent on the retry-sibling skip path and on the updateMany-count=0 path — the eventual retry's own terminal outcome is audited by its own catch. Pair this pattern with `SYSTEM_ADMIN_EMAIL` for system-initiated audit writes. Reuse when any future Tier 2 compensating tx needs to coexist with Trigger.dev-style retry semantics (multiple `CampaignDeploy` rows can exist per `Campaign`; retry lifecycle is orthogonal to rollback).

[2026-04-15T15:00:00Z] -- Pattern: CLI + HTTP route sharing via extracted helper. For any operation that needs both an HTTP surface (portal) AND a CLI surface (ops + agents), the implementation sits in `src/lib/<domain>/<op>.ts` and both callers delegate to it. The route maps helper failure codes back to HTTP status codes via a small mapper function (e.g. `deployFailureHttpStatus`). First use: BL-061 `initiateCampaignDeploy` in `src/lib/campaigns/deploy-campaign.ts`, consumed by `src/app/api/campaigns/[id]/deploy/route.ts` (non-retry branch) and `scripts/cli/campaign-deploy.ts`. The helper returns a discriminated union `{ok:true, ...} | {ok:false, code, reason, ...}` — never throws for business failures, only for genuinely unexpected errors. Dry-run is a first-class helper arg that short-circuits after validation so callers get "what would happen" without mutations. This pattern should be reused for any future CLI-wraps-route work (e.g. campaign pause/resume, sender rotation).

[2026-04-16T18:30:00Z] -- Pattern: signature-region-only text rewrites in HTML-ish email bodies. When a transform must rewrite literal text at a structurally-specific location (e.g. sender name in the signature) WITHOUT touching accidental matches mid-body (e.g. a recipient greeting that shares the sender's first name), constrain the rewrite to the "last N non-empty lines" region rather than running a global regex. First use: BL-100 `transformSenderNames` in src/lib/emailbison/sender-name-transform.ts. Key rules: (1) EB-stored email bodies use `<br>` / `<br/>` / `<br />` as line separators, NOT `\n` — a transformer that only splits on `\n` will see a single long line and either match nothing or match everywhere. Split on `(<br\s*\/?>|\n)` with a CAPTURING GROUP so the separators are preserved in the split array and reassembly is lossless (just `.join("")`). (2) Identify the signature region by walking line fragments from the end and accumulating non-empty (after-trim) entries until N is reached — N=5 empirically handles typical signature blocks (closer, name, title, phone, email) without catching closing paragraphs. (3) Match on full-line-equality-after-trim rather than `.includes()` or global regex — "Regards, Daniel" must NOT match because the full line is "Regards, Daniel" not "Daniel". Protects against multi-word lines that happen to contain the target. (4) Prefer longer tokens first when multiple could match the same line (full-name before first-name: "Daniel Lazarus" must become `{SENDER_FULL_NAME}`, not `{SENDER_FIRST_NAME} Lazarus`). (5) Idempotency marker check: if the body already contains the post-transform tokens (`\{SENDER_(?:FIRST_NAME|FULL_NAME)\}`), suppress the "no-match" warn — that's a legitimate second-pass no-op, not a writer drift. (6) Empty-roster / empty-body paths return `{transformed: body, matched: false}` silently — the caller's upstream validation (e.g. zero-senders error) is the single point of truth, not duplicate-logged here. Paired test pattern: co-locate tests that exercise BOTH plain-text `\n` bodies (unit) AND real-world `<br>` bodies (regression, pulled verbatim from production EB). Reuse pattern for any future vendor-edge transform that depends on structural body regions (unsubscribe-footer rewrites, tracking-pixel strips, disclaimer-block suppression).
