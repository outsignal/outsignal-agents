# Monty -- Architecture Patterns

<!-- Append entries as: [ISO date] -- [pattern or observation] -->

[2026-04-16T12:45:00Z] -- Pattern: findFirst-inside-tx for retry-aware atomic rollback. Any compensating transaction whose behaviour depends on the absence of a sibling row MUST run the `findFirst` inside the same `$transaction` as the write, not outside it. Reading the sibling status on the outer client and then opening a tx leaves a race hole where a second actor can enqueue the missing sibling between the read and the write. First use: BL-075 `executeDeploy` atomic Campaign.status rollback in `src/lib/campaigns/deploy.ts` — the outer catch opens a `$transaction`, inside which it (1) writes `CampaignDeploy.status='failed'` guarded by `currentDeploy.status==='running'`, (2) runs `tx.campaignDeploy.findFirst({where:{campaignId, status:{in:['pending','running']}, id:{not:deployId}}})` to check for an inflight retry sibling, (3) if absent, snapshots `Campaign.emailBisonCampaignId`, runs `tx.campaign.updateMany({where:{id,status:'deployed'},data:{status:'approved',emailBisonCampaignId:null,deployedAt:null}})` (race-safe optimistic-update pattern from `deploy-campaign.ts:130-133`), and (4) writes a `tx.auditLog.create` row. The audit is silent on the retry-sibling skip path and on the updateMany-count=0 path — the eventual retry's own terminal outcome is audited by its own catch. Pair this pattern with `SYSTEM_ADMIN_EMAIL` for system-initiated audit writes. Reuse when any future Tier 2 compensating tx needs to coexist with Trigger.dev-style retry semantics (multiple `CampaignDeploy` rows can exist per `Campaign`; retry lifecycle is orthogonal to rollback).

[2026-04-15T15:00:00Z] -- Pattern: CLI + HTTP route sharing via extracted helper. For any operation that needs both an HTTP surface (portal) AND a CLI surface (ops + agents), the implementation sits in `src/lib/<domain>/<op>.ts` and both callers delegate to it. The route maps helper failure codes back to HTTP status codes via a small mapper function (e.g. `deployFailureHttpStatus`). First use: BL-061 `initiateCampaignDeploy` in `src/lib/campaigns/deploy-campaign.ts`, consumed by `src/app/api/campaigns/[id]/deploy/route.ts` (non-retry branch) and `scripts/cli/campaign-deploy.ts`. The helper returns a discriminated union `{ok:true, ...} | {ok:false, code, reason, ...}` — never throws for business failures, only for genuinely unexpected errors. Dry-run is a first-class helper arg that short-circuits after validation so callers get "what would happen" without mutations. This pattern should be reused for any future CLI-wraps-route work (e.g. campaign pause/resume, sender rotation).
