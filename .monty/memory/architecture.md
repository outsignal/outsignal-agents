[2026-04-16T10:25:00Z] -- Pattern: shared narrowing helper for cross-class error semantics. When a single domain throws multiple Error subclasses that share a semantic (e.g. "record not found" expressed as both HTTP-404 and 200-with-empty-data), extract an `isXxx(e: unknown): boolean` helper that tests each class with instanceof + property checks. Callers use `if (isXxx(err)) { /* domain branch */ } else { throw err; }` rather than narrowing to one class and missing the other. First use: BL-078 `isNotFoundError` in `src/lib/emailbison/errors.ts`, covering `EmailBisonApiError` (HTTP 404, has `.status + .isRecordNotFound` getter) AND `EmailBisonError` (200-with-empty-data during EB's async DELETE queue window, has `.code + .statusCode`). Rules: (1) return plain `boolean`, not a type predicate â€” predicate narrowing would falsely imply the error is one of N specific classes and block legitimate access to class-specific properties (e.g. `.body` for logging) in the branch that used `instanceof` directly. (2) Never match on duck-typed plain objects (`{status: 404}`) â€” only match real instanceof. (3) Keep the helper file free of downstream imports: if class A lives in `client.ts` and class B in `types.ts`, the helper lives in its own `errors.ts` that imports from both, so `client.ts` can't import the helper back without refactoring class locations (circular dep). Internal methods on the class that throws can't use their own helper â€” callers external to the domain can. (4) Cover non-Error inputs (null/undefined/string/plain-object/symbol/number) explicitly in unit tests â€” `instanceof` returns false for all of them but the test matrix makes the contract explicit to future readers. Reuse pattern for any cross-class domain semantic (auth errors, rate-limit errors, permission-denied errors that get thrown as different classes by different transport layers).

[2026-04-16T12:45:00Z] -- Pattern: findFirst-inside-tx for retry-aware atomic rollback. Any compensating transaction whose behaviour depends on the absence of a sibling row MUST run the `findFirst` inside the same `$transaction` as the write, not outside it. Reading the sibling status on the outer client and then opening a tx leaves a race hole where a second actor can enqueue the missing sibling between the read and the write. First use: BL-075 `executeDeploy` atomic Campaign.status rollback in `src/lib/campaigns/deploy.ts` â€” the outer catch opens a `$transaction`, inside which it (1) writes `CampaignDeploy.status='failed'` guarded by `currentDeploy.status==='running'`, (2) runs `tx.campaignDeploy.findFirst({where:{campaignId, status:{in:['pending','running']}, id:{not:deployId}}})` to check for an inflight retry sibling, (3) if absent, snapshots `Campaign.emailBisonCampaignId`, runs `tx.campaign.updateMany({where:{id,status:'deployed'},data:{status:'approved',emailBisonCampaignId:null,deployedAt:null}})` (race-safe optimistic-update pattern from `deploy-campaign.ts:130-133`), and (4) writes a `tx.auditLog.create` row. The audit is silent on the retry-sibling skip path and on the updateMany-count=0 path â€” the eventual retry's own terminal outcome is audited by its own catch. Pair this pattern with `SYSTEM_ADMIN_EMAIL` for system-initiated audit writes. Reuse when any future Tier 2 compensating tx needs to coexist with Trigger.dev-style retry semantics (multiple `CampaignDeploy` rows can exist per `Campaign`; retry lifecycle is orthogonal to rollback).

[2026-04-15T15:00:00Z] -- Pattern: CLI + HTTP route sharing via extracted helper. For any operation that needs both an HTTP surface (portal) AND a CLI surface (ops + agents), the implementation sits in `src/lib/<domain>/<op>.ts` and both callers delegate to it. The route maps helper failure codes back to HTTP status codes via a small mapper function (e.g. `deployFailureHttpStatus`). First use: BL-061 `initiateCampaignDeploy` in `src/lib/campaigns/deploy-campaign.ts`, consumed by `src/app/api/campaigns/[id]/deploy/route.ts` (non-retry branch) and `scripts/cli/campaign-deploy.ts`. The helper returns a discriminated union `{ok:true, ...} | {ok:false, code, reason, ...}` â€” never throws for business failures, only for genuinely unexpected errors. Dry-run is a first-class helper arg that short-circuits after validation so callers get "what would happen" without mutations. This pattern should be reused for any future CLI-wraps-route work (e.g. campaign pause/resume, sender rotation).
