[2026-04-15T16:45:00Z] -- INCIDENT (critical): BL-061 campaign-deploy CLI creates duplicate EB campaigns instead of reusing/transitioning existing ones. Triggered 2026-04-15 12:17-12:18 UTC when the new `scripts/cli/campaign-deploy.ts` was run against 17 campaigns (7 lime-recruitment + 10 1210-solutions). For each of the 11 email campaigns the deploy helper called `EmailBisonClient.createCampaign()` TWICE (first attempt + retry), producing 22 orphan EB drafts. All 22 subsequently failed mid-upload with 422 "The title field is..." on the sequence step POST, leaving the DB campaign stuck in status='deployed' but with a 'failed' CampaignDeploy record and zero-lead/zero-sender EB debris. Root cause: the deploy path has no "reuse existing emailBisonCampaignId if present" branch — every invocation (including retry within the same CLI run) calls createCampaign afresh. Trigger for recurrence: ANY future campaign-deploy run that retries a failed deploy WILL reproduce the bug until BL-062 lands. Impact: user-visible duplicates in the EB UI, DB campaigns stuck in 'deployed' status blocking re-approval flow, wasted EB campaign IDs (numeric sequence burned). Cleanup: 22 EB drafts deleted via `src/lib/emailbison/client.ts::deleteCampaign`; 11 Campaign rows flipped deployed→approved; 11 CampaignDeploy rows flipped failed→rolled_back. Secondary observation: 4 of the 5 1210-solutions LinkedIn deploys in the same batch failed with `prisma.campaignSequenceRule.createMany()` schema errors — separate from the EB duplication bug, needs its own investigation (possible BL-062 companion item). Classification: critical until BL-062 ships (blocks safe re-runs of the CLI).
